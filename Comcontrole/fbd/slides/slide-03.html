Por que um SGBD precisa de camadas de abstração
O problema que a arquitetura em camadas resolve

Na Semana 2, vimos que um dos grandes ganhos do SGBD sobre o sistema de arquivos tradicional é a independência de dados: a possibilidade de mudar como os dados são fisicamente armazenados sem quebrar os programas que os utilizam. Mas como, na prática, um SGBD entrega essa promessa?

A resposta é uma arquitetura organizada em níveis de abstração — camadas que separam "o que o usuário vê", "como os dados são logicamente organizados" e "como os dados são fisicamente guardados no disco". Essa ideia foi formalizada em 1975 por um comitê da ANSI (American National Standards Institute), através do grupo SPARC (Standards Planning and Requirements Committee), dando origem ao que hoje chamamos de arquitetura ANSI/SPARC de três esquemas.

Analogia útil: pense em como você usa um aplicativo de banco no celular. Você vê saldo, extrato e um botão de transferência (isso é uma visão personalizada e simplificada). Por trás, o banco mantém uma estrutura lógica completa de contas, clientes e transações (o modelo lógico completo). E, ainda mais no fundo, esses dados estão gravados em servidores, discos e índices otimizados para performance (a estrutura física). Você nunca precisa saber onde fisicamente seu saldo está armazenado — e é exatamente essa separação que a arquitetura ANSI/SPARC formaliza.

Múltiplos usuários, uma única fonte de dados

Um mesmo banco de dados corporativo costuma ser usado por públicos completamente diferentes, cada um enxergando apenas o que é relevante para sua função:

  • Um vendedor só precisa ver nome do cliente, histórico de compras e limite de crédito.
  • Um analista financeiro precisa ver totais consolidados de faturamento por região.
  • Um desenvolvedor de sistemas de IA precisa de acesso amplo às tabelas brutas para treinar modelos.

Se cada um desses públicos exigisse uma cópia separada e independente dos dados, voltaríamos exatamente aos problemas de redundância e inconsistência que os sistemas de arquivo tinham (Semana 2). A arquitetura em camadas resolve isso permitindo múltiplas visões diferentes sobre uma única fonte de dados real.

A arquitetura ANSI/SPARC: os três níveis em detalhe
Os três esquemas: externo, conceitual e interno

A arquitetura ANSI/SPARC organiza um SGBD em três níveis, cada um descrevendo os mesmos dados sob uma perspectiva diferente. É importante entender que não são três cópias dos dados — são três formas diferentes de descrever um único conjunto de dados armazenado.

Nível Externo (Visões)
o que o usuário vê
Descreve a parte do banco de dados relevante para um usuário ou aplicação específica. Um mesmo banco pode ter várias visões externas diferentes simultaneamente — cada uma mostrando um subconjunto de dados, com nomes e formatos adaptados a quem vai usá-la.
Definido a partir das necessidades de cada grupo de usuários ou aplicação.
Nível Conceitual (Lógico)
estrutura lógica completa
Descreve toda a estrutura lógica do banco de dados para a comunidade de usuários como um todo: quais entidades existem, quais atributos elas têm, como se relacionam, e quais restrições de integridade se aplicam. É a "planta baixa" completa do banco — independente de como qualquer usuário específico vai enxergá-lo, e independente de como os dados são fisicamente gravados.
Definido e mantido pelo DBA, geralmente representado por um Modelo Entidade-Relacionamento.
Nível Interno (Físico)
como fica no disco
Descreve como os dados estão fisicamente armazenados: estruturas de arquivos, índices, compactação, formas de acesso, alocação de espaço em disco. É o nível mais próximo do hardware — o único que se preocupa com performance de leitura e escrita em termos concretos de armazenamento.
Gerenciado pelo DBA em conjunto com o próprio SGBD, muitas vezes de forma automática.
Pegadinha comum de prova: o nível conceitual não é "o meio-termo visual" entre o externo e o interno — ele é a descrição lógica completa do banco, da qual o nível externo extrai subconjuntos, e que o nível interno implementa fisicamente. Os três níveis descrevem os mesmos dados, sob três perspectivas diferentes, não três bancos diferentes.
Exemplo aplicado — sistema de biblioteca

Para fixar a diferença entre os três níveis, veja como cada um trataria o mesmo dado — o empréstimo de um livro — em um sistema de biblioteca:

NívelComo o empréstimo aparece
Externo (visão do bibliotecário no balcão)Uma tela simples mostrando: nome do leitor, título do livro, data de devolução — sem nenhum campo técnico irrelevante para essa tarefa.
Conceitual (modelo lógico completo)Uma entidade Emprestimo relacionando as entidades Leitor e Exemplar, com atributos como data_emprestimo, data_prevista_devolucao, status, e a restrição de que um exemplar não pode estar emprestado a dois leitores ao mesmo tempo.
Interno (armazenamento físico)Uma estrutura de arquivo em disco, com um índice B-tree sobre o campo leitor_id para acelerar a busca de "quais livros este leitor tem emprestados agora".

Note que é o mesmo empréstimo sendo descrito três vezes, de três formas — nunca três empréstimos diferentes armazenados em três lugares.

Independência de dados, mapeamentos e o papel do DBA
Independência lógica × independência física de dados

A grande razão de existir da arquitetura em três níveis é permitir que cada nível mude sem obrigar os níveis vizinhos a mudarem junto. Esse benefício tem um nome técnico específico para cada fronteira entre níveis:

Independência lógica de dados

Capacidade de alterar o nível conceitual (ex.: adicionar uma nova entidade, um novo atributo) sem precisar reescrever as aplicações que usam as visões externas — desde que essas visões específicas não sejam diretamente afetadas pela mudança.

Independência física de dados

Capacidade de alterar o nível interno (ex.: trocar um índice, reorganizar arquivos, migrar de disco) sem precisar alterar o nível conceitual nem as aplicações — porque a estrutura lógica permanece exatamente a mesma.

Qual das duas é mais fácil de alcançar na prática? A independência física, porque mudanças no armazenamento físico raramente afetam a lógica dos dados. A independência lógica é mais delicada: adicionar um atributo geralmente é seguro, mas remover ou renomear um atributo existente pode quebrar visões externas que dependiam dele — por isso mudanças na estrutura lógica de bancos em produção costumam ser tratadas com bastante cautela.
Mapeamentos entre níveis — como o SGBD faz a tradução

Para que essa separação funcione, o SGBD mantém mapeamentos (regras de correspondência) entre os níveis adjacentes:

  • Mapeamento externo/conceitual: define como cada visão externa é derivada do esquema conceitual completo — por exemplo, quais colunas de quais entidades aparecem na tela do bibliotecário.
  • Mapeamento conceitual/interno: define como cada entidade e atributo do esquema conceitual corresponde a uma estrutura física real — em qual arquivo, com qual índice, em qual formato de armazenamento.

Quando você faz uma consulta usando uma visão externa, o SGBD percorre esses mapeamentos automaticamente: traduz a visão externa para o esquema conceitual e, deste, para a estrutura física real onde os dados estão gravados — tudo de forma transparente para quem está consultando.

Fluxo simplificado de uma consulta
// Usuário, no nível EXTERNO, pede:
"mostrar nome e nota do aluno Pedro"

// SGBD traduz para o nível CONCEITUAL:
SELECT aluno.nome, matricula.nota
FROM aluno JOIN matricula ON ...
WHERE aluno.nome = 'Pedro'

// SGBD traduz para o nível INTERNO:
Acessar bloco de disco X, usar índice idx_aluno_nome,
ler os registros correspondentes, retornar ao usuário
O papel do DBA em cada nível
NívelResponsabilidade típica do DBA
ExternoCriar e manter visões específicas para diferentes grupos de usuários, controlando o que cada perfil pode enxergar.
ConceitualProjetar e evoluir o modelo lógico completo do banco — entidades, relacionamentos, restrições de integridade — geralmente junto com analistas e desenvolvedores.
InternoDefinir estruturas físicas de armazenamento, criar índices, ajustar parâmetros de performance, planejar backup e recuperação.

Essa divisão de responsabilidades ajuda a entender por que, em equipes maiores, é comum ter profissionais com focos diferentes — alguém mais próximo da modelagem de dados (nível conceitual) e alguém mais próximo da infraestrutura e performance (nível interno), mesmo que ambos usem o título genérico de "DBA".

Três níveis, um único banco

Externo (visões de usuário), conceitual (modelo lógico completo) e interno (armazenamento físico) descrevem os mesmos dados sob perspectivas diferentes.

Independência de dados

Lógica: mudar o nível conceitual sem quebrar aplicações. Física: mudar o armazenamento sem afetar a estrutura lógica.

Mapeamentos automáticos

O SGBD traduz, de forma transparente, uma consulta do nível externo até o nível interno, através de mapeamentos entre camadas.
