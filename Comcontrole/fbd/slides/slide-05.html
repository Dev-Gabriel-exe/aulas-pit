Esquema — a planta estrutural do banco de dados
Retomando o fio da meada: três semanas de contexto convergindo em um só conceito

Vamos fazer uma pausa antes de avançar, porque esta aula amarra diretamente três coisas que você já estudou. Na Semana 2, vimos por que surgiram os SGBDs: para acabar com a dependência entre programas e a forma física de armazenar dados. Na Semana 3, vimos que essa independência é obtida através de três níveis de arquitetura — interno, conceitual e externo — cada um descrevendo o banco de dados de um jeito diferente. Na Semana 4, vimos que a forma como esses níveis descrevem a estrutura dos dados pode seguir modelos completamente diferentes: árvore, grafo ou tabela.

Hoje vamos dar nome técnico e preciso a essa "descrição da estrutura": ela se chama esquema (em inglês, schema). E vamos colocar ao lado dela dois conceitos que, juntos, formam o vocabulário fundamental que qualquer profissional de banco de dados usa no dia a dia — sem exceção — para se comunicar sobre sistemas: esquema, instância e metadado. Dominar essa distinção com precisão não é só "decorar termos" — é o que separa quem entende de banco de dados de quem apenas decorou comandos SQL sem entender o que está por trás deles.

Definindo esquema: estrutura, não conteúdo

Esquema é a descrição completa da estrutura de um banco de dados: quais tabelas (ou segmentos, ou registros, dependendo do modelo estudado na Semana 4) existem, quais colunas cada uma possui, qual o tipo de dado de cada coluna, quais restrições se aplicam (por exemplo, "este campo não pode ser nulo", "este campo deve ser único"), e como as tabelas se relacionam entre si. O ponto mais importante para gravar agora é este: o esquema descreve a forma dos dados, nunca os dados em si.

Uma analogia que costuma grudar bem: pense na planta arquitetônica de uma casa. A planta especifica que existe uma sala, três quartos, dois banheiros, e onde ficam as portas e janelas. Ela não diz que existe um sofá azul na sala ou que a cama do quarto 2 está desarrumada — isso é o conteúdo da casa, não a sua estrutura. Um banco de dados funciona exatamente assim: o esquema é a planta; os dados propriamente ditos (que veremos na Aula 2, sob o nome de instância) são a casa mobiliada e habitada.

Se você já programou antes — e isso vale especialmente para quem está cursando IA junto com programação — existe uma analogia ainda mais direta: o esquema está para o banco de dados assim como uma classe está para um objeto em orientação a objetos. A classe Aluno define que todo aluno tem nome, matrícula e curso — mas a classe em si não representa nenhum aluno específico. Cada objeto instanciado a partir dela (Maria, João, Pedro) é que carrega valores reais para esses atributos. Guarde essa analogia — ela vai reaparecer na Aula 2 quando formalizarmos o conceito de instância.

Um banco de dados tem, na verdade, três esquemas — não um só

Aqui está a conexão mais importante desta aula com o que você já estudou na Semana 3: quando falamos de "o esquema" de um banco de dados de forma solta, estamos simplificando. Na prática, seguindo exatamente a mesma lógica da arquitetura em três níveis (ANSI/SPARC), um banco de dados possui três esquemas diferentes, um para cada nível — porque cada nível descreve a estrutura dos dados sob uma perspectiva distinta:

EsquemaCorresponde ao nível (Sem. 3)O que descreveQuem normalmente enxerga
Esquema internoNível interno (físico)Como os dados estão de fato armazenados em disco: arquivos, índices, métodos de acesso, blocos de armazenamento.DBA e o próprio SGBD
Esquema conceitualNível conceitual (lógico)A estrutura lógica completa do banco inteiro: todas as tabelas, colunas, tipos, restrições e relacionamentos — independente de como estão fisicamente armazenados.Projetistas de banco de dados, desenvolvedores
Esquema externoNível externo (visões)Um recorte parcial e personalizado do esquema conceitual, voltado a um grupo específico de usuários ou aplicações (o que, na prática, chamamos de view).Usuários finais, aplicações específicas

Note como essa tabela é, essencialmente, a Semana 3 revisitada com um vocabulário mais afiado: o que antes chamávamos apenas de "nível" agora ganha um nome técnico mais específico quando estamos falando da descrição estrutural daquele nível — o esquema correspondente a ele. Um mesmo banco de dados físico, portanto, pode ter um único esquema interno, um único esquema conceitual, mas vários esquemas externos diferentes — um para cada grupo de usuários com necessidades distintas.

O esquema em código: quando você "declara" uma estrutura

Você ainda não formalizou a sintaxe do CREATE TABLE — isso vem com força a partir da Semana 7, quando estudarmos a conversão do modelo Entidade-Relacionamento em tabelas. Mas vale antecipar aqui, mesmo que de forma superficial, porque ajuda a tornar o conceito de esquema muito mais concreto: quando um comando como o abaixo é executado, o que está de fato acontecendo é a declaração de um esquema — nenhum dado real foi inserido ainda, apenas a estrutura foi definida.

SQL — exemplo antecipado (formalizaremos a sintaxe completa na Sem. 7-8)
-- Isto é a DECLARAÇÃO DE UM ESQUEMA: define a estrutura,
-- não insere nenhum dado ainda.
CREATE TABLE Aluno (
    matricula   INT PRIMARY KEY,
    nome        VARCHAR(100) NOT NULL,
    curso       VARCHAR(60),
    data_nasc   DATE
);

-- Depois deste comando, o esquema "Aluno" existe --
-- mas a tabela está vazia. Isso é o assunto da próxima aula.

Perceba que essa declaração especifica exatamente os elementos que definimos como parte de um esquema: o nome da tabela, os nomes e tipos de cada coluna, e as restrições aplicáveis (PRIMARY KEY, NOT NULL). Nenhuma linha desse comando fala sobre "quem" são os alunos — isso é, propositalmente, o assunto da Aula 2.

O esquema como contrato: estabilidade, governança e o papel do DBA

Um esquema não é definido uma única vez e esquecido — mas também não é algo que muda com frequência, e essa estabilidade relativa é proposital. Pense no esquema como uma espécie de contrato entre o banco de dados e todos os programas que dependem dele: enquanto o esquema permanece o mesmo, qualquer aplicação que sabe "conversar" com aquela estrutura continua funcionando sem qualquer alteração. Foi exatamente essa estabilidade que discutimos, na Semana 3, como o objetivo central por trás da independência de dados.

Alterar um esquema em produção — adicionar uma coluna, mudar o tipo de um campo, remover uma tabela — é uma operação delicada, chamada de evolução de esquema (schema evolution), e normalmente exige planejamento cuidadoso: é preciso avaliar qual impacto a mudança terá sobre todos os programas e relatórios que já dependem da estrutura antiga. É justamente por esse motivo que a definição, alteração e manutenção do esquema costuma ser responsabilidade formal do DBA (Administrador de Banco de Dados) — papel que vamos estudar com mais profundidade na Semana 9.

Por que isso importa na prática profissional: em sistemas de longa duração — um banco usado por um hospital ou por uma prefeitura ao longo de décadas, por exemplo — o esquema tende a evoluir lentamente e de forma controlada, exatamente porque cada mudança de estrutura tem o potencial de quebrar sistemas inteiros construídos em cima dele. É uma das razões pelas quais bancos de dados relacionais bem projetados são tão duradouros: o esquema, uma vez estabilizado, raramente precisa de alterações drásticas.

Instância — o banco de dados em um instante do tempo
Definindo instância: o conteúdo real, capturado em um momento

Se o esquema é a planta da casa, a instância é a casa mobiliada, habitada e fotografada em um instante específico do tempo. Formalmente: instância (também chamada de estado do banco de dados, ou ocorrência) é o conjunto real de dados armazenados no banco de dados em um determinado momento, respeitando a estrutura definida pelo esquema. Retomando a analogia de programação da Aula 1: se o esquema é a classe, a instância é o conjunto de todos os objetos criados a partir dela, com seus valores reais preenchidos.

Voltando ao nosso CREATE TABLE Aluno da aula passada: assim que inserimos dados reais nessa tabela, criamos uma instância. Veja a diferença lado a lado:

A tabela de cima nunca muda enquanto o esquema estiver estável. A tabela de baixo muda a cada novo aluno matriculado, a cada troca de curso, a cada atualização de dados — potencialmente milhares de vezes por dia em um sistema real. Essa diferença de frequência de mudança é, na prática, a característica mais marcante que distingue esquema de instância.

Instância como fotografia: o banco de dados muda de estado a cada transação

Um jeito preciso de pensar sobre instância é como uma fotografia tirada em um instante específico. Cada operação de INSERT, UPDATE ou DELETE executada sobre o banco de dados produz uma nova instância — tecnicamente, o banco de dados "transiciona" de um estado para o próximo. Isso significa que, ao longo do tempo, um único esquema pode corresponder a uma sequência praticamente infinita de instâncias diferentes: a instância no momento da matrícula de Maria é diferente da instância um segundo depois, quando João também se matricula.

Instância t₀
Tabela recém-criada, vazia
Zero linhas. O esquema já existe (a estrutura está definida), mas a instância nesse momento é o conjunto vazio.
Instância t₁
INSERT de Maria
A instância passa a conter uma linha. O esquema permanece exatamente o mesmo de antes.
Instância t₂
INSERT de João e Pedro
A instância agora contém três linhas. Ainda o mesmo esquema.
Instância t₃
UPDATE do curso de João
A instância mudou de novo — o valor da coluna "curso" na linha do João foi alterado. Continua o mesmo esquema.

Esse encadeamento de instâncias sucessivas é o que faz um banco de dados ser, na prática, um sistema profundamente dinâmico por dentro de uma estrutura relativamente estática. É também a base conceitual para entender, mais adiante no curso, tópicos como transações (uma transação nada mais é do que uma transição controlada e segura de uma instância para outra) e backup/recuperação (que veremos na Semana 14): fazer um backup é, essencialmente, guardar uma cópia de uma instância específica em um determinado instante, para poder restaurá-la depois caso algo dê errado.

Instância válida × instância inválida: o papel das restrições do esquema

Aqui está um ponto sutil, mas muito importante: nem toda combinação possível de dados é uma instância válida para um determinado esquema. O esquema, através das suas restrições, define exatamente quais estados a instância tem permissão de assumir. Se tentarmos inserir dados que violem essas restrições, o SGBD deve rejeitar a operação — a instância "proibida" nunca chega a existir de fato.

Instância válida

Uma nova linha inserida com matricula = 2026004, nome = 'Ana Souza', respeitando o tipo INT da matrícula, o tipo texto do nome, e sem repetir uma matrícula já existente (respeitando a PRIMARY KEY).

Instância inválida (rejeitada pelo SGBD)

Uma tentativa de inserir uma linha com nome = NULL, violando a restrição NOT NULL definida no esquema; ou uma matrícula repetida, violando a PRIMARY KEY. O SGBD recusa a operação antes mesmo dela se tornar uma instância real.

Essa ideia — de que o esquema atua como um "filtro de validade" para qualquer instância possível — é a semente de um assunto que vamos estudar com profundidade na Semana 11: integridade de dados. Por ora, o importante é entender que esquema e instância não são conceitos independentes: o esquema restringe quais instâncias são permitidas, e é justamente essa restrição que impede que o banco de dados acumule dados incoerentes ou corrompidos ao longo do tempo.

Metadados e o catálogo de dados do SGBD
O que são metadados: "dados sobre os dados"

Vimos até agora que o esquema descreve a estrutura, e a instância é o conteúdo real. Mas surge uma pergunta natural: onde, exatamente, o SGBD guarda a descrição do esquema? Afinal, se o esquema diz "a tabela Aluno tem uma coluna nome do tipo VARCHAR(100)", essa própria afirmação também precisa estar armazenada em algum lugar, de alguma forma, dentro do sistema. É aqui que entra o conceito de metadado.

Metadado significa, literalmente, "dado sobre o dado". É a informação que descreve outros dados: sua estrutura, seu significado, sua origem, suas restrições e seu contexto de uso. O esquema de um banco de dados, na verdade, é armazenado internamente pelo SGBD como metadado — ou seja, o esquema não é uma abstração "fora" do banco de dados, mas sim um conjunto de dados especial, guardado pelo próprio sistema, que descreve a estrutura de todos os outros dados.

Uma forma simples de fixar a diferença: se a instância é "os dados sobre os alunos" (nomes, matrículas, cursos), o metadado é "os dados sobre a tabela Aluno" (quantas colunas ela tem, quais os tipos, quais restrições existem). É informação em uma camada acima — dados descrevendo a própria estrutura de armazenamento, e não o mundo real que os dados representam.
Três tipos de metadado que convivem em qualquer SGBD

Na prática, "metadado" é um guarda-chuva que cobre pelo menos três categorias diferentes de informação, todas armazenadas e gerenciadas pelo SGBD:

  • Metadados estruturais: a definição do próprio esquema — nomes de tabelas, colunas, tipos de dados, chaves primárias e estrangeiras, índices. É a categoria que estudamos na Aula 1.
  • Metadados descritivos/administrativos: informações sobre o histórico e o contexto dos dados — quem criou uma tabela, em que data, quando foi a última modificação de estrutura, qual o tamanho ocupado em disco, estatísticas de uso.
  • Metadados de segurança e acesso: quais usuários têm permissão de ler, escrever ou alterar cada tabela — um tema que vamos aprofundar na Semana 12, quando estudarmos consistência e segurança de dados.

Um jeito prático de perceber que essas três categorias realmente existem: qualquer SGBD relacional moderno permite que você consulte, através de comandos SQL comuns, informações sobre as próprias tabelas do sistema — coisas como "quais tabelas existem neste banco?" ou "quais colunas a tabela Aluno possui?". Essas perguntas não retornam dados de alunos: elas retornam metadados sobre a estrutura Aluno em si.

O catálogo de dados (data dictionary / system catalog): onde o SGBD guarda a si mesmo

O lugar onde um SGBD armazena todos os seus metadados tem nome próprio: catálogo de dados, também chamado de dicionário de dados (data dictionary) ou catálogo do sistema (system catalog). É, na prática, um "banco de dados dentro do banco de dados" — uma coleção de tabelas internas, mantidas e atualizadas automaticamente pelo próprio SGBD, que descrevem exaustivamente toda a estrutura de todas as outras tabelas do sistema.

Essa ideia — de que um SGBD guarda a descrição de si mesmo usando o próprio modelo relacional — foi uma inovação relevante do projeto System R, o protótipo desenvolvido pela IBM nos anos 1970 para provar que o modelo relacional proposto por Codd (que estudamos na Semana 4) era, de fato, implementável na prática. A ideia de um banco de dados autodescritivo — capaz de armazenar sua própria estrutura usando o mesmo mecanismo que armazena os dados dos usuários — se tornou um padrão praticamente universal em todos os SGBDs relacionais modernos.

SQL — consultando o catálogo do sistema (padrão INFORMATION_SCHEMA)
-- "INFORMATION_SCHEMA" é um catálogo de metadados
-- presente, com pequenas variações, em praticamente
-- todo SGBD relacional (PostgreSQL, MySQL, SQL Server...)

-- Pergunta: quais tabelas existem neste banco de dados?
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'public';

-- Pergunta: quais colunas a tabela "Aluno" possui,
-- e qual o tipo de cada uma?
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_name = 'aluno';

-- Repare: o RESULTADO dessas consultas não é
-- "dados de alunos" -- é METADADO sobre a estrutura.

Esse exemplo antecipa, de forma bem concreta, algo que vamos formalizar apenas nas Semanas 15 e 16 (linguagens de definição e manipulação de dados) — mas o objetivo aqui não é ensinar a sintaxe SQL em profundidade, e sim mostrar que o catálogo de dados não é um conceito abstrato: é uma estrutura real, consultável, que existe dentro de todo SGBD relacional que você vai usar na sua carreira.

Do dicionário de dados ao catálogo de dados corporativo

Em sistemas pequenos, o catálogo de dados interno do SGBD já é suficiente. Mas em organizações que lidam com dezenas ou centenas de bancos de dados diferentes — múltiplos sistemas, plataformas de nuvem, data warehouses e data lakes coexistindo — surge a necessidade de um nível adicional de organização: o catálogo de dados corporativo (enterprise data catalog). Diferente do catálogo interno de um único SGBD, essa ferramenta centraliza os metadados de múltiplas fontes de dados diferentes em um só lugar, permitindo que qualquer pessoa da organização descubra "onde estão os dados que eu preciso" sem precisar conhecer cada sistema individualmente.

Ferramentas como AWS Glue Data Catalog, Alation, Amundsen (criada originalmente pelo Lyft) e DataHub (criada pelo LinkedIn) são exemplos reais dessa categoria, amplamente usados em times de dados e IA no mercado atual. Elas não guardam os dados propriamente ditos — guardam metadados sobre onde os dados estão, o que significam, quem é o responsável por eles, e com que frequência são atualizados.

Olhando à frente: catálogos de dados corporativos são a espinha dorsal do que chamamos de governança de dados — um tema que ganha peso crescente à medida que times de IA precisam confiar na qualidade, origem e significado dos dados usados para treinar modelos. Você vai reencontrar essa ideia, com muito mais profundidade, em disciplinas futuras voltadas a Big Data e engenharia de dados.
