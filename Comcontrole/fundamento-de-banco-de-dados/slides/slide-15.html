O que é DDL, e onde ela se encaixa dentro do SQL
SQL não é uma linguagem só — são quatro, disfarçadas de uma

Ao longo do semestre, usamos comandos SQL em diversos contextos: escrevemos CREATE TABLE na Semana 8 para representar o modelo lógico, usamos GRANT e REVOKE na Semana 9 para controlar acesso, e mencionamos pg_dump na Semana 14 ao falar de backup. Chegou o momento de organizar tudo isso: o SQL (Structured Query Language) não é uma linguagem única e homogênea — ele se divide, tradicionalmente, em quatro subfamílias de comandos, cada uma com uma responsabilidade bem definida.

SubfamíliaO que controlaComandos principaisOnde já vimos (ou veremos)
DDL
Data Definition Language		A estrutura do banco: tabelas, colunas, restrições, índicesCREATE, ALTER, DROP, TRUNCATEUsada informalmente na Semana 8; formalizada hoje
DML
Data Manipulation Language		Os dados que vivem dentro da estrutura já criadaINSERT, SELECT, UPDATE, DELETEAssunto completo da Semana 16
DCL
Data Control Language		Quem tem permissão para acessar o quêGRANT, REVOKEEstudada na Semana 9
TCL
Transaction Control Language		O agrupamento de várias operações em uma única unidade tudo-ou-nadaCOMMIT, ROLLBACKMencionada brevemente hoje; aprofundada em disciplinas futuras de banco de dados avançado
Por que essa divisão existe: ela reflete, quase perfeitamente, os próprios papéis que estudamos na Semana 9 — o DBA concentra a maior parte dos privilégios de DDL e DCL (estrutura e permissões), enquanto desenvolvedores e analistas vivem, no dia a dia, principalmente no mundo da DML (os dados em si). Entender essa separação ajuda a prever, mesmo antes de conhecer a sintaxe exata, qual comando pertence a qual "caixa de ferramentas".
CREATE — dando o primeiro passo: criando um banco de dados

O comando mais fundamental de toda a DDL é o CREATE, usado para trazer à existência um novo objeto dentro do SGBD — seja um banco de dados inteiro, uma tabela, um índice ou outros objetos que veremos ao longo do curso. Vamos começar pelo nível mais alto, o próprio banco de dados:

SQL — criando um banco de dados novo
-- Cria um banco de dados vazio, ainda sem nenhuma tabela:
CREATE DATABASE faculdade_pit;

-- A partir daqui, todas as tabelas que criarmos
-- (Aluno, Curso, Matricula...) vivem DENTRO deste banco.

Um banco de dados, nesse sentido, funciona como um "container" organizacional: ele isola completamente os objetos de um sistema dos objetos de outro sistema, mesmo que ambos rodem no mesmo servidor físico e sejam administrados pelo mesmo SGBD.

CREATE TABLE, agora com sintaxe completa e tipos de dado formais

Já usamos CREATE TABLE exaustivamente na Semana 8, mas sempre de forma um pouco informal, focando na lógica de chaves e cardinalidade. Hoje formalizamos um detalhe que deixamos em aberto: os tipos de dado que cada coluna pode receber. Assim como uma variável em uma linguagem de programação, toda coluna de uma tabela relacional precisa declarar um tipo de dado específico, que determina o que pode (e o que não pode) ser armazenado nela.

Tipo de dadoArmazenaExemplo de uso
INT / INTEGERNúmeros inteirosMatrícula, código, quantidade
DECIMAL(p,s) / NUMERIC(p,s)Números com casas decimais exatas (sem erro de arredondamento)Preço, salário, valores monetários
VARCHAR(n)Texto de tamanho variável, até n caracteresNome, e-mail, endereço
TEXTTexto de tamanho livre, sem limite práticoDescrição longa, comentário
DATEApenas data (dia, mês, ano)Data de nascimento, data de matrícula
TIMESTAMPData e hora combinadasMomento exato de um cadastro ou de uma transação
BOOLEANVerdadeiro ou falsoAluno está ativo? Curso está disponível?
SQL — CREATE TABLE completo, com tipos de dado formais
CREATE TABLE Curso (
    codigo        INT PRIMARY KEY,
    nome_curso    VARCHAR(80) NOT NULL,
    carga_horaria INT,
    ativo         BOOLEAN DEFAULT TRUE,
    data_criacao  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
Restrições (constraints): reforçando as regras diretamente na estrutura

Perceba, no exemplo acima, que colocamos algumas palavras além do tipo de dado — NOT NULL, DEFAULT TRUE. Essas são restrições (constraints): regras que o próprio SGBD passa a impor automaticamente, rejeitando qualquer comando que tente violá-las. Isso conecta diretamente com o conceito de integridade que estudamos na Semana 11 — ali discutimos os tipos de integridade (domínio, entidade, referencial); hoje vemos, na prática, a sintaxe que implementa cada um deles.

RestriçãoO que garanteExemplo
PRIMARY KEYIdentificação única de cada linha (integridade de entidade, Semana 11)codigo INT PRIMARY KEY
FOREIGN KEYReferência válida a outra tabela (integridade referencial, Semanas 8 e 11)FOREIGN KEY (curso_codigo) REFERENCES Curso(codigo)
NOT NULLA coluna nunca pode ficar vazianome_curso VARCHAR(80) NOT NULL
UNIQUENenhum valor pode se repetir na coluna, mesmo sem ser chave primáriaemail VARCHAR(120) UNIQUE
DEFAULTValor assumido automaticamente quando nenhum é informadoativo BOOLEAN DEFAULT TRUE
CHECKUma condição lógica que todo valor precisa satisfazer (integridade de domínio, Semana 11)CHECK (carga_horaria > 0)
Exercício 01 — Escrevendo os primeiros CREATE TABLE com constraints

Escreva o comando CREATE TABLE completo para uma entidade Produto de um sistema de estoque, respeitando as seguintes regras de negócio:

(a) Todo produto tem um código numérico único, que é a chave primária.
(b) O nome do produto é obrigatório (não pode ficar vazio).
(c) O preço precisa ser um valor decimal, e nunca pode ser menor ou igual a zero.
(d) Todo produto novo entra automaticamente como "disponível" no estoque, a menos que seja informado o contrário.
(e) O código de barras (um texto de até 20 caracteres) não pode se repetir entre produtos diferentes, mesmo não sendo a chave primária.

ALTER — evoluindo uma estrutura já existente sem perder dados
Por que a estrutura de um banco de dados muda com o tempo

Nenhum sistema real nasce pronto e permanece exatamente igual para sempre. Novas funcionalidades exigem novos dados; regras de negócio mudam; erros de modelagem aparecem apenas depois que o sistema já está em uso, com dados reais dentro dele. Recriar a tabela inteira do zero toda vez que algo muda — apagando e digitando tudo de novo — não é uma opção realista quando já existem milhares (ou milhões) de linhas de dados dentro dela. É exatamente para isso que existe o comando ALTER TABLE: ele permite evoluir a estrutura de uma tabela já existente, preservando os dados que já estão lá dentro.

As variações do ALTER TABLE, uma a uma

O ALTER TABLE não é um único comando fixo — é uma família de operações diferentes, todas começando pela mesma cláusula inicial, mas seguidas de instruções distintas dependendo do que se deseja mudar:

SQL — variações de ALTER TABLE sobre a tabela Aluno (Semana 8)
-- Adicionando uma nova coluna:
ALTER TABLE Aluno ADD COLUMN email VARCHAR(120);

-- Removendo uma coluna que não é mais necessária:
ALTER TABLE Aluno DROP COLUMN data_nasc;

-- Renomeando uma coluna existente:
ALTER TABLE Aluno RENAME COLUMN nome TO nome_completo;

-- Alterando o TIPO de dado de uma coluna já existente:
ALTER TABLE Aluno ALTER COLUMN matricula TYPE BIGINT;

-- Adicionando uma restrição a uma coluna que já existia:
ALTER TABLE Aluno ADD CONSTRAINT email_unico UNIQUE (email);

-- Removendo uma restrição que não faz mais sentido:
ALTER TABLE Aluno DROP CONSTRAINT email_unico;

-- Renomeando a própria tabela:
ALTER TABLE Aluno RENAME TO Estudante;

Repare que cada uma dessas operações resolve um problema estrutural bem específico — e que, coletivamente, elas cobrem praticamente qualquer ajuste evolutivo que um sistema real vai exigir ao longo dos anos de uso.

Os riscos escondidos de um ALTER TABLE em produção

Nem toda alteração de estrutura é "de graça" do ponto de vista técnico. Alguns comandos ALTER TABLE, especialmente em tabelas grandes já em uso por um sistema em produção, podem ter consequências sérias se executados sem planejamento:

RiscoPor que aconteceComo mitigar
Bloqueio da tabela (lock)Alguns SGBDs precisam reescrever a tabela inteira em disco ao alterar um tipo de dado ou adicionar certas restrições, bloqueando leituras e escritas durante o processo.Executar em horário de baixo tráfego; testar previamente o tempo de execução em ambiente de desenvolvimento (Semana 9).
Perda irreversível de dadosUm DROP COLUMN remove, junto com a coluna, todos os valores que já estavam armazenados nela — sem aviso adicional.Gerar um backup imediatamente antes (Semana 14) e confirmar cuidadosamente o nome exato da coluna.
Incompatibilidade de tipoAlterar uma coluna de VARCHAR para INT, por exemplo, falha se já existir algum valor de texto que não seja um número válido.Validar os dados existentes antes de tentar a conversão de tipo.
Quebra da aplicaçãoSe o desenvolvedor (Semana 9) não for avisado, o código da aplicação pode continuar referenciando uma coluna renomeada ou removida.Comunicação prévia entre DBA e equipe de desenvolvimento antes de qualquer alteração estrutural.
Pare e pense: imagine que você precisa remover a coluna cpf de uma tabela Aluno que já tem 50 mil linhas, em um sistema em uso por uma faculdade inteira. Antes mesmo de pensar na sintaxe do comando, que perguntas você faria à equipe antes de executar essa alteração?

DROP, TRUNCATE e o checklist de segurança para comandos destrutivos
DROP — o comando mais definitivo de toda a DDL

Chegamos ao comando mais perigoso — e, por isso mesmo, mais importante de entender com clareza — desta aula: DROP. Diferente do ALTER, que evolui uma estrutura preservando o que pode ser preservado, o DROP remove completamente um objeto do banco de dados: sua estrutura e todos os dados que ele contém, de uma vez só, sem pedir confirmação adicional e, na maioria dos SGBDs, sem nenhuma forma nativa de desfazer a ação depois.

SQL — comandos DROP
-- Remove a tabela inteira, estrutura E dados, para sempre:
DROP TABLE Matricula;

-- Evita erro caso a tabela já não exista
-- (útil em scripts que podem ser executados mais de uma vez):
DROP TABLE IF EXISTS Matricula;

-- Remove o banco de dados inteiro -- TODAS as tabelas dentro dele:
DROP DATABASE faculdade_pit;
Não existe "Ctrl+Z" para um DROP em produção sem backup. É exatamente por isso que a Semana 14 não foi um assunto isolado: todo comando destrutivo de DDL deveria, na prática profissional, pressupor a existência de um backup recente e testado — a rede de segurança que permite que um erro humano seja um contratempo, e não uma tragédia.
DROP × TRUNCATE × DELETE: três formas de "esvaziar" uma tabela

É extremamente comum confundir três comandos que, à primeira vista, parecem fazer a mesma coisa — "apagar dados" — mas que têm naturezas, custos e reversibilidades completamente diferentes. Vale a pena fixar essa comparação, porque ela também antecipa o DELETE, comando de DML que veremos formalmente na Semana 16:

ComandoSublinguagemO que removePode ter WHERE?Reversível?
DROP TABLEDDLA estrutura inteira e todos os dadosNãoNão (sem backup)
TRUNCATE TABLEDDLTodos os dados, mas mantém a estrutura da tabela intactaNãoNão (sem backup)
DELETE FROMDMLApenas as linhas especificadas (ou todas, se não houver filtro)SimSim, se dentro de uma transação não confirmada (TCL)

Note a diferença sutil, mas importante, entre TRUNCATE e DELETE sem WHERE: ambos removem todas as linhas de uma tabela, mas o TRUNCATE normalmente executa de forma muito mais rápida, porque não processa linha por linha nem gera o mesmo tipo de registro individual no log de transações que estudamos na Semana 14 — o que também significa que ele costuma ser mais difícil (ou impossível) de reverter dentro de uma transação em andamento.

O checklist de segurança antes de qualquer comando DDL destrutivo

Juntando tudo o que discutimos nas três aulas de hoje com o que já vimos nas Semanas 9 e 14, chegamos a um checklist prático — o tipo de rotina que equipes de dados maduras seguem, quase por reflexo, antes de rodar qualquer ALTER, DROP ou TRUNCATE em um ambiente de produção:

  • Gerar um backup recente imediatamente antes da operação (Semana 14) — mesmo que já exista uma rotina automática, uma cópia extra "momentos antes" reduz o RPO ao mínimo possível.
  • Testar o comando em ambiente de desenvolvimento primeiro, com uma cópia dos dados, para confirmar que ele faz exatamente o que deveria (Semana 9).
  • Pedir revisão de uma segunda pessoa antes de rodar em produção — a mesma lógica de separação de funções discutida na Semana 9.
  • Escolher um horário de baixo tráfego para minimizar o impacto de um eventual bloqueio de tabela.
  • Conferir duas vezes o nome exato da tabela, coluna ou banco de dados alvo — um erro de digitação em um comando DROP não avisa antes de executar.
Exemplo guiado completo: o ciclo de vida de uma tabela em DDL

Para fechar a semana, vamos aplicar CREATE, ALTER e DROP em sequência, sobre um cenário completo e continuando com as tabelas que já conhecemos desde a Semana 8 — simulando, de forma resumida, como uma estrutura de banco de dados realmente evolui ao longo do tempo:

SQL — ciclo de vida completo: criar, evoluir e descontinuar
-- Fase 1: CREATE -- a tabela nasce, simples e enxuta:
CREATE TABLE Disciplina (
    codigo INT PRIMARY KEY,
    nome   VARCHAR(80) NOT NULL
);

-- Fase 2: ALTER -- o sistema evolui, e a tabela evolui junto:
ALTER TABLE Disciplina ADD COLUMN carga_horaria INT;
ALTER TABLE Disciplina ADD CONSTRAINT ch_positiva CHECK (carga_horaria > 0);
ALTER TABLE Disciplina ADD COLUMN ativa BOOLEAN DEFAULT TRUE;

-- Fase 3: alguns meses depois, um campo antigo
-- se torna obsoleto e é removido (após backup e revisão):
ALTER TABLE Disciplina DROP COLUMN ativa;

-- Fase 4: DROP -- se a disciplina inteira for descontinuada
-- do currículo (com todos os dados já migrados ou arquivados):
DROP TABLE IF EXISTS Disciplina;

Esse é, resumidamente, o ciclo de vida real de praticamente qualquer tabela em um sistema de longa duração: ela nasce com CREATE, passa por múltiplas rodadas de ALTER ao longo dos anos conforme o sistema cresce, e — eventualmente, em alguns casos — é removida com DROP quando deixa de fazer sentido, sempre seguindo o checklist de segurança que acabamos de estudar.

O que ainda falta — e onde vamos chegar em breve

Hoje aprendemos a criar e evoluir a estrutura vazia de um banco de dados — as "prateleiras", mas ainda sem nenhum produto nelas. A pergunta natural que fica em aberto é: como, então, colocamos e manipulamos os dados dentro dessa estrutura? É exatamente esse o assunto da Semana 16: a linguagem de manipulação de dados (DML) — INSERT, SELECT, UPDATE e DELETE — o complemento natural de tudo o que construímos hoje.
