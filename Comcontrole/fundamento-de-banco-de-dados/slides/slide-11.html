Integridade de dados: as três garantias fundamentais
O que significa, de fato, "integridade de dados"

Integridade de dados é a garantia de que os dados armazenados em um banco são válidos, corretos e obedecem às regras estruturais definidas no seu esquema. Diferente do que se possa imaginar, integridade não é um conceito vago de "qualidade dos dados" — é algo bem concreto, imposto pelo próprio SGBD através de restrições (constraints) declaradas junto com as tabelas.

A literatura de banco de dados costuma dividir a integridade em três categorias bem definidas. Vamos ver cada uma com exemplos práticos, retomando as tabelas Aluno, Curso e Matricula que vimos construindo desde a Semana 7.

Tipo de integridadeO que garanteMecanismo SQL típico
Integridade de domínioCada valor armazenado em uma coluna pertence ao conjunto de valores válidos para aquele campo (tipo, formato, faixa)CHECK, tipo de dado da coluna, NOT NULL
Integridade de entidadeCada linha de uma tabela é unicamente identificável — nenhuma chave primária se repete ou fica vaziaPRIMARY KEY
Integridade referencialToda referência entre tabelas aponta para um registro que realmente existeFOREIGN KEY
Integridade de domínio: valores dentro do esperado

A integridade de domínio garante que uma coluna nunca receba um valor sem sentido para o que ela representa — uma idade negativa, uma nota de prova de 15 em uma escala de 0 a 10, um e-mail sem "@". Ela é imposta principalmente através do tipo de dado da coluna e de restrições CHECK explícitas.

SQL — restrições de domínio na tabela Matricula
CREATE TABLE Matricula (
    id INT PRIMARY KEY,
    aluno_id INT NOT NULL,
    disciplina_id INT NOT NULL,
    nota DECIMAL(4,2) CHECK (nota >= 0 AND nota <= 10),
    status VARCHAR(20) CHECK (status IN ('cursando', 'aprovado', 'reprovado')),
    data_matricula DATE NOT NULL
);

-- Tentativas que a integridade de domínio bloqueia automaticamente:
INSERT INTO Matricula (id, aluno_id, disciplina_id, nota, status, data_matricula)
VALUES (1, 10, 3, 15.0, 'aprovado', '2025-03-01');
-- ERRO: nota 15.0 viola o CHECK (nota <= 10)

INSERT INTO Matricula (id, aluno_id, disciplina_id, nota, status, data_matricula)
VALUES (2, 10, 3, 8.5, 'matriculado', '2025-03-01');
-- ERRO: 'matriculado' não está na lista de status permitidos

Note que essas validações acontecem dentro do próprio SGBD, e não dependem de nenhum programa aplicativo lembrar de checá-las — exatamente o ganho de "integridade centralizada" que apontamos, lá na Semana 2, como uma das razões de existir dos SGBDs.

Integridade de entidade: cada linha, um identificador único

A integridade de entidade garante duas coisas simultaneamente sobre a chave primária de uma tabela: ela nunca pode ser NULL, e ela nunca pode se repetir entre duas linhas diferentes. Sem essa garantia, seria impossível apontar, de forma confiável, para "um" registro específico — a própria noção de "linha única" desmoronaria.

SQL — integridade de entidade em ação
CREATE TABLE Aluno (
    matricula VARCHAR(20) PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    email VARCHAR(100)
);

INSERT INTO Aluno (matricula, nome) VALUES ('2024001234', 'Ana Souza');
INSERT INTO Aluno (matricula, nome) VALUES ('2024001234', 'Outro Aluno');
-- ERRO: matrícula '2024001234' já existe (viola PRIMARY KEY)

INSERT INTO Aluno (matricula, nome) VALUES (NULL, 'Aluno Sem Matrícula');
-- ERRO: chave primária não pode ser NULL
Por que isso conecta direto com a Semana 8: quando escolhemos uma chave primária ao modelar uma tabela, estamos, na prática, decidindo qual coluna (ou conjunto de colunas) vai carregar a responsabilidade de garantir a integridade de entidade daquela tabela. Uma escolha ruim de chave primária — por exemplo, usar nome como chave em vez de matricula — compromete diretamente essa garantia, já que nomes podem se repetir entre pessoas diferentes.
Integridade referencial: toda referência precisa "existir de verdade"

É a mais rica das três, porque envolve duas tabelas ao mesmo tempo. A integridade referencial garante que uma chave estrangeira (FOREIGN KEY) só aponte para valores que realmente existem como chave primária na tabela referenciada — nunca para um registro "fantasma".

SQL — integridade referencial entre Matricula e Aluno
CREATE TABLE Matricula (
    id INT PRIMARY KEY,
    aluno_id VARCHAR(20) NOT NULL,
    disciplina_id INT NOT NULL,
    FOREIGN KEY (aluno_id) REFERENCES Aluno(matricula)
);

INSERT INTO Matricula (id, aluno_id, disciplina_id) VALUES (1, '9999999999', 3);
-- ERRO: não existe nenhum aluno com matrícula '9999999999'
-- A FOREIGN KEY impede a criação dessa "referência fantasma"

A integridade referencial também define o que fazer quando um registro referenciado é removido — e esse comportamento precisa ser decidido explicitamente pelo DBA ou modelador, através das cláusulas ON DELETE:

CláusulaComportamento ao apagar o registro "pai"Exemplo de uso
ON DELETE CASCADEApaga automaticamente também os registros filhosApagar um Curso apaga suas Disciplinas associadas
ON DELETE RESTRICTImpede a exclusão do "pai" enquanto existirem "filhos" ligados a eleImpedir apagar um Aluno que ainda tem Matriculas ativas
ON DELETE SET NULLZera a referência nos registros filhos, sem apagá-losApagar um Professor deixa a Disciplina "sem professor" em vez de apagá-la

Consistência de dados: além do que o SGBD consegue checar sozinho
Integridade × consistência: uma distinção que cai em prova

É comum confundir integridade com consistência, mas a diferença é importante — e frequentemente cobrada em avaliações. Integridade, como vimos na Aula 1, é sobre obedecer às restrições estruturais declaradas no esquema (tipos, chaves, relacionamentos). Consistência é um conceito mais amplo: é a garantia de que o banco de dados, como um todo, reflete fielmente as regras de negócio do mundo real que ele representa — mesmo quando essas regras vão além do que um simples CHECK ou FOREIGN KEY consegue expressar.

Integridade

Pode ser garantida inteiramente pelo SGBD, de forma automática, através de restrições declaradas na estrutura das tabelas.

Consistência

Muitas vezes depende de regras de negócio complexas que envolvem múltiplas tabelas, cálculos ou lógica que vai além do que uma simples restrição consegue expressar sozinha.

Um jeito simples de lembrar a diferença: todo dado inconsistente com as regras de negócio quebra alguma expectativa do sistema, mas nem todo dado inconsistente necessariamente viola uma restrição de integridade formal — porque a regra de negócio pode ser complexa demais para caber em um CHECK simples.
Exemplo: um dado que é íntegro, mas inconsistente

Considere o cenário de um sistema acadêmico onde a regra de negócio diz: "um aluno só pode se matricular em uma disciplina se já tiver cursado e sido aprovado nos pré-requisitos dela". Suponha o seguinte INSERT:

SQL — passa nas restrições de integridade, mas quebra a regra de negócio
-- Ana (matrícula '2024001234') tenta se matricular em
-- "Banco de Dados II", cujo pré-requisito é "Banco de Dados I" —
-- mas Ana nunca cursou "Banco de Dados I".
INSERT INTO Matricula (id, aluno_id, disciplina_id, status, data_matricula)
VALUES (55, '2024001234', 12, 'cursando', '2025-08-01');

-- Esse INSERT passa perfeitamente por todas as restrições de integridade:
-- • a chave primária 'id=55' é única (integridade de entidade) ✓
-- • aluno_id '2024001234' existe na tabela Aluno (integridade referencial) ✓
-- • disciplina_id '12' existe na tabela Disciplina (integridade referencial) ✓
-- • status 'cursando' está entre os valores permitidos (integridade de domínio) ✓
--
-- Mas o dado é INCONSISTENTE: viola uma regra de negócio que
-- nenhuma FOREIGN KEY ou CHECK simples consegue expressar sozinha.

Esse é exatamente o ponto central da Aula 2: toda violação de integridade gera inconsistência, mas nem toda inconsistência é capturada como violação de integridade. Regras de negócio mais elaboradas — que dependem de consultar o histórico de outras tabelas, ou de lógica condicional mais complexa — costumam exigir mecanismos adicionais, como triggers (gatilhos automáticos) ou validações na camada de aplicação, temas que outras disciplinas do curso (como Desenvolvimento Front-end para IA) vão aprofundar.

Mais exemplos de inconsistência de negócio
CenárioPor que é uma inconsistênciaPor que uma restrição simples não resolve sozinha
Uma disciplina tem carga horária de 60h, mas a soma das cargas horárias de suas turmas cadastradas soma 45hOs dados relacionados não "fecham" entre siExige comparar valores agregados de tabelas diferentes, não um único valor isolado
Um aluno aparece como "aprovado" em uma disciplina, mas sua nota registrada é 4,0 (abaixo da média de aprovação, 6,0)O status contradiz o dado numérico relacionadoDepende de comparar duas colunas da mesma linha sob uma regra específica de negócio (pode até ser resolvido com CHECK envolvendo as duas colunas, mas exige planejamento deliberado)
Um professor está cadastrado como responsável por duas turmas com horários que se sobrepõemFisicamente impossível de cumprir na realidadeExige comparar horários entre múltiplas linhas de uma mesma tabela
Vale reforçar: nenhum desses três exemplos quebra uma PRIMARY KEY, uma FOREIGN KEY ou um CHECK simples — todos passariam despercebidos por uma verificação de integridade básica. É exatamente por isso que garantir consistência, na prática, costuma exigir mais do que apenas um bom desenho de restrições: exige processos, revisões e, em muitos casos, lógica adicional dentro ou fora do banco.
Exercício 02 — Integridade ou consistência?

Para cada situação, diga se ela representa uma violação de integridade (capturável por uma restrição simples do SGBD) ou de consistência de negócio (que exige lógica adicional), justificando:

Recuperação de dados: o que acontece quando algo dá errado
Por que falhas são inevitáveis — e por que isso não é motivo de pânico

Por mais bem projetado que um banco de dados seja em termos de integridade e consistência, ele continua rodando sobre hardware real, energia elétrica real e software escrito por seres humanos — e todos esses elementos podem falhar. Recuperação de dados é a capacidade de um SGBD retornar a um estado correto e consistente depois que algum tipo de falha acontece, sem perder (ou perdendo o mínimo possível) do trabalho já realizado.

Tipo de falhaExemplo concretoO que normalmente é afetado
Falha de transaçãoUma operação viola uma restrição no meio da execução, ou o programa aplicativo travaApenas a operação em andamento — o restante do banco permanece intacto
Falha de sistemaQueda de energia, travamento do sistema operacional, reinicialização inesperada do servidorDados que ainda estavam apenas em memória, não gravados em disco
Falha de mídiaO disco físico onde o banco está armazenado é danificado ou corrompidoPotencialmente todo o conteúdo gravado naquele disco

Cada uma dessas falhas exige uma estratégia de recuperação diferente — mas todas elas dependem de um mesmo mecanismo central, que vamos detalhar a seguir: o log de transações.

O log de transações: a "caixa-preta" do banco de dados

A grande maioria dos SGBDs relacionais modernos mantém um log de transações (também chamado de write-ahead log, ou WAL) — um registro sequencial e contínuo de todas as operações de escrita realizadas no banco, gravado antes mesmo de a alteração definitiva acontecer nos dados propriamente ditos.

Representação simplificada de um log de transações
-- Cada linha do log registra uma operação, com identificador
-- de transação, para permitir desfazer ou refazer se necessário:

[T101] INICIO
[T101] INSERT Aluno (matricula='2024005678', nome='Bruno Lima')
[T101] COMMIT

[T102] INICIO
[T102] UPDATE Matricula SET nota=8.5 WHERE id=203
[T102] UPDATE Matricula SET status='aprovado' WHERE id=203
-- ↑ T102 ainda não deu COMMIT quando o sistema caiu.
--   O log permite que o SGBD saiba exatamente onde parar
--   e o que precisa ser desfeito na recuperação.

Quando o SGBD é reiniciado após uma falha, ele passa por um processo automático chamado recuperação por log (log-based recovery), que segue, em essência, duas ações complementares:

REDO (refazer)

Reaplica operações de transações que já haviam confirmado (COMMIT) no log, mas cujo efeito ainda não havia sido gravado fisicamente nos arquivos de dados no momento da falha.

UNDO (desfazer)

Reverte qualquer efeito parcial de transações que estavam em andamento, mas não haviam confirmado (COMMIT) antes da falha — como a transação T102 do exemplo acima.

Por que isso garante consistência mesmo após uma queda de energia: como o log é escrito antes da alteração definitiva (daí o nome write-ahead), o SGBD sempre tem um registro confiável de exatamente onde parou — permitindo reconstruir com precisão o que deveria (REDO) e o que não deveria (UNDO) ter permanecido no banco.
Backup: a rede de segurança para além do log

O log de transações resolve muito bem falhas de sistema (quedas de energia, travamentos) — mas não resolve uma falha de mídia, onde o próprio disco físico é danificado e o log também pode ser perdido junto. Para esse cenário, a estratégia complementar é o backup: uma cópia completa (ou parcial) dos dados, armazenada de forma separada e segura.

  • Backup completo: uma cópia integral de todo o banco de dados em um determinado momento — mais simples de restaurar, porém mais lento e pesado de gerar.
  • Backup incremental: registra apenas o que mudou desde o último backup — mais rápido de gerar, mas a restauração exige aplicar vários incrementos em sequência.
  • Combinação com o log: um backup completo periódico, somado ao log de transações contínuo, permite restaurar o banco para qualquer ponto exato no tempo — não apenas para o momento do último backup.
Isso é só o começo do assunto: hoje vimos o conceito e a lógica geral de backup e recuperação. A Semana 14 (nas disciplinas de banco de dados mais avançadas do curso) aprofunda estratégias completas de backup, recuperação de desastres e alta disponibilidade — aqui, o objetivo é entender por que essas estratégias existem antes de aprender a executá-las tecnicamente.
Linha do tempo de uma falha e recuperação — passo a passo
Passo 1
Operação normal
Transações acontecem normalmente; cada operação é registrada no log de transações antes de ser efetivada nos dados.
Passo 2
A falha ocorre
Por exemplo, uma queda de energia interrompe o sistema no meio de uma transação que ainda não confirmou (COMMIT).
Passo 3
Sistema reinicia e lê o log
O SGBD, ao voltar, examina o log de transações para identificar exatamente o que estava em andamento e o que já havia confirmado.
Passo 4
REDO e UNDO são aplicados
Transações confirmadas, mas não persistidas, são refeitas (REDO); transações não confirmadas são desfeitas (UNDO).
Passo 5
Banco volta a um estado consistente
O sistema retoma operação normal, com garantia de que nenhuma transação ficou "pela metade" — a mesma ideia de atomicidade que retomaremos, formalmente, quando estudarmos transações em profundidade em disciplinas futuras.
