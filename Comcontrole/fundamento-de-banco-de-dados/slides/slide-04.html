O modelo hierárquico — dados organizados em árvore
Retomando o contexto: por que precisamos de "modelos" de dados

Na Semana 2 vimos quando e por que surgiram os primeiros SGBDs. Mas dizer "existe um software central gerenciando os dados" não responde a uma pergunta fundamental: de que forma esse software organiza, internamente, os dados que ele gerencia? Essa é exatamente a pergunta que um modelo de dados responde — um modelo de dados é o conjunto de conceitos usados para descrever a estrutura de um banco de dados: quais são as unidades básicas de organização, como elas se relacionam entre si, e quais operações são permitidas sobre elas.

Entre o final dos anos 1960 e o início dos anos 1970, coexistiam basicamente três abordagens concorrentes para responder a essa pergunta: o modelo hierárquico, o modelo em rede e o modelo relacional. Hoje vamos estudar os dois primeiros em profundidade — o hierárquico nesta aula, o de rede na próxima — para, na terceira aula, entender com clareza técnica por que o relacional se tornou dominante e permanece assim até hoje.

Por que estudar modelos "antigos" em 2026? Porque eles não desapareceram — reapareceram. Bancos de documentos como o MongoDB organizam dados de forma essencialmente hierárquica (documentos aninhados dentro de documentos). Bancos de grafos como o Neo4j retomam ideias muito próximas do modelo em rede. Entender a lógica original ajuda a entender essas tecnologias modernas com muito mais profundidade do que apenas "decorar comandos".
Estrutura do modelo hierárquico

No modelo hierárquico, os dados são organizados como uma árvore: existe um único registro no topo (a raiz), e cada registro abaixo dele é um filho de exatamente um pai. Essa é a regra mais importante do modelo — e também sua maior limitação, como veremos adiante: todo registro tem no máximo um pai. Tecnicamente, cada tipo de registro é chamado de segmento, e a relação entre um segmento pai e um segmento filho é sempre do tipo 1:N (um pai pode ter vários filhos, mas cada filho pertence a um único pai).

Vamos visualizar isso com um cenário familiar ao contexto acadêmico — a estrutura de uma faculdade:

Faculdade PIT
Curso: Tecnólogo em IA
Curso: Bacharelado em IA
Turma BD1.02-A
Turma BD1.02-B
Aluno: Maria
Aluno: João
Aluno: Pedro

Note que cada Turma pertence a exatamente um Curso, e cada Aluno, nesse desenho, pertence a exatamente uma Turma. Essa estrutura funciona perfeitamente enquanto a realidade que estamos modelando também for, de fato, estritamente hierárquica. O problema — que vamos explorar em detalhe já já — é que a realidade raramente é tão organizada assim.

Como se acessa um dado: navegação, não consulta

Essa é talvez a diferença mais importante entre o modelo hierárquico e o modelo relacional que você já está começando a estudar: não existe uma linguagem de consulta declarativa como o SQL. Para chegar a um registro específico, o programa precisa navegar fisicamente pela árvore, começando sempre pela raiz e descendo nível por nível, seguindo ponteiros gravados fisicamente entre os registros.

Pseudocódigo — acesso hierárquico (estilo IMS/DL-I)
// Para encontrar "todas as notas do aluno Maria",
// o programa PRECISA descer pela árvore nesta ordem:

GET UNIQUE Faculdade (WHERE nome = 'Faculdade PIT')
GET NEXT   Curso     (WHERE nome = 'Tecnólogo em IA')
GET NEXT   Turma      (WHERE codigo = 'BD1.02-A')
GET NEXT   Aluno       (WHERE nome = 'Maria')
GET NEXT   Nota

// Não existe "buscar direto pela Maria" — é obrigatório
// percorrer cada nível pai antes de alcançar o filho.

Esse estilo de acesso é chamado de navegacional ou procedural: o programador precisa descrever o caminho exato até o dado (o "como chegar"), e não apenas o que ele quer (que é o que o SQL, décadas depois, tornaria possível). Isso tem uma consequência séria: se a estrutura da árvore mudar — por exemplo, se um novo nível for inserido entre Curso e Turma — todo programa que navegava por aquele caminho pode parar de funcionar. Lembra da independência de dados que estudamos na Semana 3? O modelo hierárquico, na prática, viola exatamente esse princípio: a estrutura física de navegação está embutida na lógica dos programas.

Contexto histórico: o IMS e o programa Apollo

O exemplo mais famoso de SGBD hierárquico é o IMS (Information Management System), desenvolvido pela IBM em 1966 — já mencionado na nossa linha do tempo da Semana 2. A IBM precisava gerenciar a imensa lista de peças (e sub-peças, e sub-sub-peças) de um foguete Saturno V para o programa espacial Apollo: cada peça "contém" outras peças, numa estrutura naturalmente hierárquica — exatamente o tipo de cenário em que esse modelo brilha.

O IMS nunca morreu de verdade. Diferente de outros sistemas da mesma geração, o IMS continua em produção até hoje em bancos e seguradoras que processam volumes massivos de transações (muitos caixas eletrônicos e sistemas de cartão de crédito no mundo ainda dependem, direta ou indiretamente, de mainframes rodando IMS). Isso acontece porque, para o caso de uso específico para o qual foi desenhado — hierarquias profundas, acesso previsível, altíssima performance — ele continua extremamente eficiente.
A limitação central: relacionamentos N:N

Aqui está o problema estrutural mais grave do modelo hierárquico, e é fundamental entendê-lo bem, porque ele é exatamente o tipo de problema que motivou a busca por um modelo melhor. Pense num cenário muito comum: um Aluno pode se matricular em várias Turmas, e uma Turma tem vários Alunos. Esse é um relacionamento N:N (muitos-para-muitos) — e o modelo hierárquico simplesmente não consegue representar isso de forma nativa, porque sua regra fundamental é "todo filho tem exatamente um pai".

Para contornar essa limitação, existiam basicamente duas soluções — e ambas são ruins:

Solução 1 — Duplicar o registro

Se a Maria está matriculada em duas turmas, o registro completo da Maria (nome, CPF, endereço...) é fisicamente duplicado sob cada turma. Isso reintroduz exatamente o problema de redundância e inconsistência que vimos na Semana 2 — se o endereço da Maria muda, alguém precisa lembrar de atualizar as duas cópias.

Solução 2 — "Filhos lógicos" / ponteiros extras

SGBDs hierárquicos mais avançados (como o próprio IMS) permitem simular um segundo "pai" através de ponteiros adicionais entre árvores separadas. Isso funciona, mas exige um esforço de projeto e programação consideravelmente maior — na prática, é o sistema forçando manualmente uma flexibilidade que o modelo, por definição, não tem.

Guarde essa ideia: a incapacidade nativa de representar relacionamentos N:N é o problema que, mais adiante no semestre (Semanas 7 e 8, ao estudarmos o modelo Entidade-Relacionamento e sua conversão para tabelas), veremos ser resolvido de forma elegante pelo modelo relacional através de tabelas associativas com chaves estrangeiras.

O modelo em rede — resolvendo o problema do "pai único"
A ideia central: um registro pode ter mais de um pai

O modelo em rede surgiu como resposta direta à limitação mais dolorosa do modelo hierárquico: a impossibilidade de um registro ter mais de um "pai". Estruturalmente, o modelo em rede é uma generalização do modelo hierárquico — em vez de uma árvore rígida, os registros formam um grafo, onde um registro filho pode estar conectado a múltiplos registros pais simultaneamente.

Voltando ao nosso exemplo de Aluno e Turma: no modelo em rede, é perfeitamente possível representar diretamente que a Maria está matriculada tanto na Turma BD1.02-A quanto em uma eventual Turma MLOps-A, sem duplicar o registro da Maria e sem recorrer a ponteiros "extras" artificiais. O relacionamento N:N deixa de ser um problema a ser contornado e passa a ser algo que o próprio modelo representa de forma nativa — decompondo-o em dois relacionamentos 1:N simultâneos.

Lembrando a Semana 2: o primeiro SGBD baseado nesse modelo foi o IDS (Integrated Data Store), criado por Charles Bachman na General Electric em 1960 — o mesmo IDS que já apareceu na nossa linha do tempo da evolução dos sistemas de armazenamento. Bachman, inclusive, recebeu o Prêmio Turing de 1973 (o "Nobel da Computação") justamente pelo seu trabalho pioneiro em bancos de dados de rede.
O padrão CODASYL e o conceito de "conjunto" (set)

O modelo em rede foi formalizado como padrão em 1969 por um grupo de trabalho chamado CODASYL DBTG (Conference on Data Systems Languages — Data Base Task Group) — o mesmo consórcio industrial que, curiosamente, também padronizou a linguagem COBOL. O padrão CODASYL introduziu um conceito técnico central chamado conjunto (set): uma estrutura que formaliza a relação entre um registro owner (proprietário/pai) e vários registros member (membros/filhos).

A diferença crucial em relação ao modelo hierárquico é que um mesmo registro pode ser membro de vários conjuntos diferentes ao mesmo tempo — e é exatamente essa flexibilidade que permite representar relacionamentos N:N de forma direta, combinando dois conjuntos:

Estrutura conceitual — dois "sets" resolvendo N:N
// Conjunto 1: TURMA (owner) → ALUNO (member)
SET Turma_tem_Alunos
  OWNER:  Turma
  MEMBER: Aluno

// Conjunto 2: ALUNO (owner) → TURMA (member)
SET Aluno_cursa_Turmas
  OWNER:  Aluno
  MEMBER: Turma

// A combinação dos dois "sets" permite que Maria
// pertença a duas turmas SEM duplicar o registro da Maria,
// e que uma turma tenha vários alunos SEM duplicar a turma.
Navegação em rede: ainda procedural, agora mais complexa

É importante não idealizar o modelo em rede: ele resolveu o problema do N:N, mas manteve — e em certo sentido, agravou — o outro grande problema do modelo hierárquico: o acesso continua sendo navegacional, não declarativo. O programador ainda precisa escrever código explícito para "andar" entre registros, usando comandos como FIND, GET e FIND NEXT WITHIN SET, sempre mantendo controle da posição atual (chamada de currency, ou "corrente") dentro de cada conjunto.

Como agora existem múltiplos caminhos possíveis entre dois registros (porque um registro pode pertencer a vários conjuntos), o programador precisa saber, de antemão, exatamente qual caminho seguir para chegar aonde quer — a complexidade de programação aumenta consideravelmente em relação ao já complicado modelo hierárquico.

O que o modelo em rede ganhou

Suporte nativo a relacionamentos N:N; maior flexibilidade estrutural; ainda muito eficiente em termos de performance para navegação predefinida, porque os ponteiros entre registros são resolvidos fisicamente, sem necessidade de "descobrir" relações em tempo de execução.

O que o modelo em rede manteve (ou piorou)

Programação procedural de baixo nível; forte acoplamento entre a estrutura física do banco e o código das aplicações; curva de aprendizado alta para os desenvolvedores; qualquer alteração na definição dos conjuntos pode quebrar programas existentes.

Os três modelos lado a lado — por que o relacional venceu
O que os modelos hierárquico e em rede têm em comum — e por que isso é um problema

Apesar das diferenças estruturais entre árvore e grafo, os modelos hierárquico e em rede compartilham três características de fundo que, juntas, formam exatamente o conjunto de problemas que o modelo relacional foi desenhado para resolver:

  • Dependência de ponteiros físicos: os relacionamentos entre registros são implementados como ponteiros gravados fisicamente no disco, não como um conceito lógico independente da implementação.
  • Acesso procedural (navegacional): não existe uma linguagem declarativa — o programador precisa descrever, passo a passo, o caminho exato até o dado desejado, em vez de apenas descrever o que quer obter.
  • Fragilidade da independência de dados: como a lógica de navegação está amarrada à estrutura física, qualquer alteração na estrutura do banco tende a quebrar programas existentes — o oposto do que estudamos como objetivo central da arquitetura ANSI/SPARC na Semana 3.

Essas limitações não eram um detalhe técnico menor: elas tornavam bancos de dados caros de manter (qualquer mudança de estrutura exigia reescrever programas) e caros de usar (qualquer pergunta nova, não prevista na estrutura de navegação original, exigia um novo programa escrito por um especialista).

A ruptura de Codd: dados como tabelas, consulta como matemática

É nesse cenário que, em 1970, o pesquisador da IBM Edgar F. Codd publica o artigo que já mencionamos na Semana 2, "A Relational Model of Data for Large Shared Data Banks". A proposta de Codd era radical para a época: abandonar completamente os ponteiros como mecanismo de relacionamento entre dados, e organizar tudo como relações — que, na prática, correspondem ao que hoje chamamos de tabelas, com linhas e colunas.

A genialidade da proposta estava em dois pontos que os modelos anteriores simplesmente não tinham:

  • Fundamentação matemática sólida: Codd baseou o modelo na teoria dos conjuntos e na lógica de predicados — áreas bem estabelecidas da matemática. Isso permitiu definir formalmente operações como seleção, projeção e junção (join), que hoje formam a base do SQL.
  • Relacionamentos por valor, não por ponteiro: em vez de um ponteiro físico ligando um registro a outro, o relacionamento passa a ser expresso por um valor compartilhado entre tabelas (o que hoje conhecemos como chave primária e chave estrangeira). Isso desacopla completamente a lógica dos dados da forma como eles são fisicamente armazenados.
A consequência mais prática dessa mudança: pela primeira vez, um usuário podia fazer uma pergunta nova ao banco de dados — algo como "quantos alunos moram em Teresina?" — sem que ninguém precisasse escrever um programa novo especificamente para essa pergunta. Bastava formular uma consulta declarativa, dizendo o que se queria, e deixar que o SGBD descobrisse como buscar essa informação.

Vale memorizar para a prova: a vantagem decisiva do modelo relacional não foi "ser mais moderno" — foi resolver, ao mesmo tempo, os três problemas estruturais compartilhados pelos modelos anteriores: dependência de ponteiros, acesso procedural e independência de dados fraca. Nenhum dos dois modelos anteriores resolvia os três de uma vez.
Um epílogo importante: os modelos antigos não desapareceram de verdade

Vale fechar esta aula com uma reflexão que vai acompanhar você pelo resto do curso: o triunfo do modelo relacional não significou o desaparecimento completo das ideias dos modelos hierárquico e em rede — elas simplesmente reapareceram, décadas depois, sob nomes diferentes, dentro do movimento que hoje chamamos de NoSQL (que você vai estudar com mais profundidade nas disciplinas de Banco de Dados avançado e Big Data):

  • Bancos de documentos (MongoDB, Couchbase) organizam dados como documentos JSON aninhados dentro de outros documentos — estruturalmente, uma forma de árvore, muito próxima do espírito do modelo hierárquico.
  • Bancos de grafos (Neo4j, Amazon Neptune) representam explicitamente nós conectados por múltiplas arestas — uma ideia estruturalmente muito próxima do modelo em rede, agora com uma linguagem de consulta declarativa muito mais amigável (como o Cypher).

Essa não é uma coincidência — é um lembrete de que, em bancos de dados, modelos antigos raramente "morrem": eles voltam quando o tipo de problema para o qual foram desenhados volta a ser relevante, geralmente com uma linguagem de consulta muito melhor do que a original.

Estudo de caso da semana: em grupo, escolham um cenário de dados do cotidiano (ex.: uma rede social, um sistema de delivery, uma biblioteca) e modelem-no tanto como uma estrutura hierárquica quanto como um esquema relacional simples (apenas na forma de tabelas, sem se preocupar com sintaxe SQL ainda). Comparem: o que ficou mais simples em cada modelo? Onde o modelo hierárquico "quebrou" diante de um relacionamento N:N?

Modelo hierárquico

Estrutura em árvore, cada filho com um único pai; forte em cenários hierárquicos, mas incapaz de representar N:N de forma nativa (IMS, IBM 1966).

Modelo em rede

Registros com múltiplos "pais" via conjuntos (sets); resolve N:N, mas mantém acesso procedural e complexidade alta (CODASYL, 1969).

Modelo relacional

Tabelas ligadas por valores (chaves), consulta declarativa e base matemática sólida — resolve os três problemas de uma vez (Codd, 1970).
