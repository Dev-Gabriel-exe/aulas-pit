O que é, exatamente, independência de dados
Retomando a arquitetura de três níveis

Na Semana 3 estudamos a arquitetura ANSI/SPARC, que organiza um SGBD em três camadas: o nível externo (as visões que cada usuário ou aplicação enxerga), o nível conceitual (a estrutura lógica completa do banco — entidades, atributos, relacionamentos) e o nível interno (como os dados realmente ficam gravados fisicamente em disco). Vale relembrar rapidamente essa organização antes de avançarmos:

Nível Externo — o que cada usuário/aplicação vê
Nível Conceitual — o modelo lógico completo do banco
Nível Interno — como os dados ficam fisicamente armazenados

A pergunta que fica em aberto depois da Semana 3 é: o que, na prática, garante que uma mudança em um desses níveis não obrigue a reescrever tudo nos níveis vizinhos? A resposta é justamente o conceito que damos nome hoje: independência de dados.

Definindo independência de dados com precisão

Independência de dados é a capacidade de alterar a definição de um esquema em um nível da arquitetura sem exigir alteração no esquema do nível imediatamente superior. É importante notar duas palavras nessa definição: capacidade e sem exigir. Independência de dados não significa que uma mudança nunca afeta nada acima — significa que o SGBD foi projetado para que, na maioria dos casos razoáveis, essa mudança fique contida no nível onde ocorreu.

Uma analogia que ajuda a fixar: pense em uma apresentação de slides que busca dados de uma planilha através de um link. Se você reorganiza as células da planilha por trás (adiciona uma nova aba, reordena linhas) mas mantém os nomes das células referenciadas, o slide continua funcionando sem que você precise tocar nele. Agora, se você deletar a célula que o slide referencia diretamente, o link quebra. A independência de dados é justamente o esforço de projetar o sistema para que o primeiro cenário (mudança "por trás" sem impacto visível) seja a regra, e o segundo (quebra direta) seja a exceção rara e evitável.

Existem exatamente dois tipos de independência de dados, cada um correspondendo a uma fronteira específica entre níveis da arquitetura ANSI/SPARC — e são eles que vamos detalhar, com bastante profundidade e exemplos práticos, nas próximas duas aulas.

TipoFronteira protegidaPergunta que ela responde
Independência físicaEntre o nível interno e o nível conceitual"Posso mudar como os dados são guardados sem mexer na estrutura lógica nem nas aplicações?"
Independência lógicaEntre o nível conceitual e o nível externo"Posso mudar a estrutura lógica do banco sem quebrar as aplicações que já o utilizam?"
Por que uma é bem mais fácil de garantir do que a outra

Um ponto que costuma gerar confusão em prova: os dois tipos de independência não são igualmente fáceis de alcançar na prática. A independência física é, de forma geral, bem mais simples de garantir de forma quase total — porque mudanças de armazenamento raramente têm qualquer relação lógica com o significado dos dados. Já a independência lógica é estruturalmente mais delicada, porque o próprio significado dos dados pode mudar junto com a estrutura.

  • Independência física — quase sempre alcançável: trocar um índice, mudar o formato de compactação de um arquivo, migrar de um disco para outro — nada disso altera o que os dados significam, então as aplicações não sentem a diferença.
  • Independência lógica — alcançável apenas parcialmente: adicionar uma nova coluna geralmente é seguro, mas remover uma coluna que uma aplicação já usa, por definição, precisa afetar essa aplicação — não existe mágica que resolva isso sem alguma adaptação.

Guarde essa assimetria — ela vai aparecer de novo, com muito mais detalhe, na Aula 3 de hoje, quando discutirmos os limites reais da independência lógica.

Independência física de dados, em profundidade
O que pertence ao "nível físico" de um banco de dados

Antes de falar sobre a independência em si, vale reforçar o que exatamente conta como parte do nível interno (físico) de um banco de dados — o alvo das mudanças que estamos discutindo nesta aula:

  • Índices — estruturas auxiliares que aceleram a busca por determinados valores (tema que será aprofundado tecnicamente na Semana 14).
  • Formato de armazenamento — como os registros são fisicamente organizados em blocos de disco, incluindo compactação.
  • Particionamento — dividir uma tabela muito grande em partes menores, fisicamente separadas, para facilitar consultas e manutenção.
  • Local de armazenamento físico — em qual disco, servidor ou até data center os dados residem.
  • Estatísticas e estruturas de otimização — informações internas que o SGBD usa para decidir a forma mais eficiente de executar uma consulta.

Nenhum desses elementos altera o que os dados significam — apenas como eles são fisicamente organizados e acessados por trás dos panos.

Exemplo guiado: adicionando um índice sem quebrar nada

Imagine que, no banco acadêmico que vimos construindo desde a Semana 7 (tabelas Aluno, Curso, Matricula), o DBA percebe que a consulta abaixo — usada com muita frequência pela secretaria — está demorando mais do que deveria conforme o número de alunos cresce:

SQL — consulta frequente da secretaria
SELECT nome, email FROM Aluno WHERE matricula = '2024001234';

Para resolver isso, o DBA decide criar um índice sobre a coluna matricula — uma mudança inteiramente contida no nível interno:

SQL — mudança no nível INTERNO
-- Antes: buscar por matrícula percorria a tabela inteira
-- Depois de criar o índice, a busca se torna muito mais rápida:
CREATE INDEX idx_aluno_matricula ON Aluno (matricula);
O que NÃO muda

A consulta da secretaria continua exatamente igual — mesmo texto SQL, mesmo resultado. A estrutura lógica da tabela Aluno (Semana 8) permanece idêntica. Nenhuma aplicação precisa ser alterada.

O que muda

Apenas a forma como o SGBD localiza fisicamente os dados no disco — agora usando o índice em vez de percorrer a tabela inteira. A consulta simplesmente fica mais rápida.

Esse é o exemplo mais clássico de independência física de dados em ação: uma decisão inteiramente de engenharia de performance, invisível para quem escreve ou usa as consultas.

Mais exemplos de mudanças fisicamente independentes
Mudança físicaMotivo comumImpacto no nível conceitual/externo
Particionar a tabela Matricula por ano letivoTabela cresceu demais e consultas ficaram lentasNenhum — SELECT * FROM Matricula WHERE ano = 2025 continua igual
Migrar os arquivos do banco de um disco HDD para SSDGanho de performance em leitura/escritaNenhum — é uma decisão puramente de infraestrutura
Comprimir os dados armazenados fisicamenteRedução de custo de armazenamentoNenhum — a leitura descomprime automaticamente, de forma transparente
Remover um índice pouco utilizadoÍndices desnecessários deixam operações de escrita mais lentasNenhum na maioria dos casos, exceto se alguma consulta específica dependia fortemente dele para performance
Atenção ao detalhe da última linha: independência física garante que a mudança não quebra a correção das consultas (elas continuam retornando o resultado certo), mas não garante, sozinha, que a performance permaneça idêntica. Remover um índice mal planejado pode deixar uma consulta específica mais lenta — sem, no entanto, alterar seu resultado nem exigir reescrevê-la.

Independência lógica de dados: alcances e limites reais
O que pertence ao "nível lógico" de um banco de dados

A independência lógica protege a fronteira entre o nível conceitual (a estrutura lógica completa: quais entidades, atributos, relacionamentos e restrições existem — construída na Semana 7 e traduzida em tabelas na Semana 8) e o nível externo (as visões específicas que cada aplicação ou usuário efetivamente utiliza). Mudanças típicas neste nível incluem: adicionar ou remover uma tabela, adicionar, remover ou renomear uma coluna, mudar um relacionamento entre entidades, ou alterar uma restrição de integridade.

Diferente da independência física, aqui o próprio significado dos dados pode estar em jogo — e é exatamente por isso que essa independência é mais difícil de garantir de forma completa.

O caso fácil: adicionar sem remover

A categoria de mudança lógica mais segura é a adição pura — criar uma nova tabela, ou adicionar uma nova coluna a uma tabela existente, sem alterar nada do que já existia.

SQL — mudança lógica ADITIVA (segura)
-- A faculdade decide passar a registrar o email institucional
-- de cada aluno. Isso é uma adição pura à tabela Aluno:
ALTER TABLE Aluno ADD COLUMN email_institucional VARCHAR(100);

-- Todas as consultas antigas continuam funcionando perfeitamente:
SELECT nome, matricula FROM Aluno WHERE curso_id = 3;
-- ↑ Nenhuma mudança necessária aqui — a coluna nova
--   simplesmente não é referenciada, e isso não é um erro.

Note que qualquer visão externa que já existia — qualquer consulta, relatório ou tela de aplicação — continua funcionando exatamente como antes, porque nenhuma delas dependia da existência da nova coluna. É este o cenário que mais se aproxima de uma independência lógica "completa" na prática.

O caso difícil: quando a independência lógica esbarra na realidade

Agora o contraponto importante, que costuma ser mal compreendido: existem mudanças lógicas que, por definição, não podem ser absorvidas sem algum tipo de impacto em quem consome os dados. Isso não é uma falha do SGBD — é uma consequência lógica inevitável de remover ou transformar algo que já estava sendo usado.

Remover uma coluna usada

Se uma aplicação faz SELECT telefone FROM Aluno e o DBA remove a coluna telefone, essa consulta simplesmente vai falhar. Não existe forma de "independência lógica" que preserve o funcionamento de uma consulta que depende de algo que deixou de existir.

Dividir uma tabela em duas

Se a tabela Aluno é dividida em Aluno e Aluno_Contato por motivos de normalização, qualquer consulta antiga que buscava telefone/endereço diretamente de Aluno precisará ser reescrita com um JOIN.

O papel real do SGBD nesses casos: mesmo quando a independência lógica "completa" é impossível, um bom SGBD oferece ferramentas para minimizar o impacto — e é exatamente disso que trata o próximo tópico desta aula.
Views: a principal ferramenta prática de independência lógica

Uma view (visão) é uma consulta salva que se comporta como se fosse uma tabela — é, literalmente, a implementação prática do nível externo da arquitetura ANSI/SPARC que estudamos na Semana 3. Views são a ferramenta mais usada no mercado para amortecer o impacto de mudanças no nível conceitual sobre as aplicações que consomem os dados.

Vamos retomar o exemplo da divisão da tabela Aluno em duas, mas agora usando uma view para proteger as aplicações antigas:

SQL — usando uma VIEW para preservar compatibilidade
-- Estrutura NOVA, já normalizada, no nível conceitual:
CREATE TABLE Aluno (
    id INT PRIMARY KEY,
    nome VARCHAR(100),
    matricula VARCHAR(20)
);

CREATE TABLE Aluno_Contato (
    aluno_id INT REFERENCES Aluno(id),
    telefone VARCHAR(20),
    endereco VARCHAR(200)
);

-- View que RECRIA a aparência da tabela antiga,
-- no nível externo, para as aplicações que ainda
-- esperam ver tudo junto:
CREATE VIEW Aluno_Legado AS
SELECT a.id, a.nome, a.matricula, c.telefone, c.endereco
FROM Aluno a
JOIN Aluno_Contato c ON a.id = c.aluno_id;

-- A aplicação antiga muda APENAS o nome da tabela na consulta,
-- sem precisar reescrever nenhuma lógica de negócio:
SELECT nome, telefone FROM Aluno_Legado WHERE matricula = '2024001234';

Esse padrão — manter uma view com a "aparência antiga" enquanto o esquema conceitual evolui por trás — é extremamente comum em sistemas reais de médio e grande porte, exatamente porque reescrever todas as aplicações de uma vez, no mesmo dia de uma mudança estrutural, costuma ser inviável ou arriscado demais.

Dois tipos de independência

Física (interno ↔ conceitual) e lógica (conceitual ↔ externo) — cada uma protege uma fronteira diferente da arquitetura ANSI/SPARC.

Física é quase sempre total

Índices, particionamento e migrações de armazenamento não afetam o significado dos dados nem quebram consultas existentes.

Lógica é parcial — e views ajudam

Adições são seguras; remoções e reestruturações exigem adaptação, frequentemente amortecida com o uso de views.
