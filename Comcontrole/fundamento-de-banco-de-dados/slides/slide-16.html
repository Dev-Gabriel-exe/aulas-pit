De prateleiras vazias a dados de verdade

Na Semana 15, fechamos o estudo da DDL com uma analogia: CREATE TABLE constrói as "prateleiras" de um banco de dados, mas ainda sem nenhum produto nelas. Hoje colocamos os produtos nas prateleiras. A DML (Data Manipulation Language) é a subfamília do SQL responsável por tudo o que acontece com os dados depois que a estrutura já existe — inserir, consultar, alterar e remover linhas dentro das tabelas que projetamos desde a Semana 8.

Vamos trabalhar, ao longo de toda a semana, com as mesmas tabelas que já são "velhas conhecidas" nossas: Curso, Aluno, Disciplina e Matricula, construídas nas Semanas 8 e 15. A vantagem de reaproveitar esse cenário é dupla: você já entende a estrutura de cor, e consegue focar inteiramente na novidade de hoje — o que fazer com os dados dentro dela.

INSERT — a porta de entrada dos dados

O comando INSERT adiciona uma (ou várias) novas linhas a uma tabela já existente. Sua sintaxe básica exige informar o nome da tabela, as colunas que serão preenchidas e os valores correspondentes, na mesma ordem:

SQL — inserindo dados nas tabelas do sistema acadêmico
-- Inserindo um curso:
INSERT INTO Curso (codigo, nome_curso, carga_horaria)
VALUES (1, 'Tecnologia em Banco de Dados', 1600);

-- Inserindo um aluno matriculado nesse curso:
INSERT INTO Aluno (matricula, nome, curso_codigo)
VALUES (2026001, 'Maria Eduarda Silva', 1);

-- Inserindo VÁRIAS linhas em um único comando,
-- mais eficiente do que repetir o INSERT várias vezes:
INSERT INTO Disciplina (codigo, nome, carga_horaria) VALUES
    (101, 'Fundamentos de Banco de Dados', 60),
    (102, 'Lógica de Programação', 80),
    (103, 'Redes de Computadores', 60);

Repare que, se uma coluna não for mencionada no INSERT, o SGBD tenta usar o valor DEFAULT definido para ela (Semana 15) — e, se não houver um DEFAULT e a coluna tiver a restrição NOT NULL, o comando inteiro falha, recusando a inserção. Esse é, na prática, o momento em que todas as restrições que definimos na semana passada realmente "entram em ação": o SGBD nunca aceita um INSERT que viole uma PRIMARY KEY duplicada, uma FOREIGN KEY inexistente ou uma condição de CHECK não satisfeita.

Conectando com a Semana 8: tente inserir um Aluno com curso_codigo = 99, sendo que nenhum curso de código 99 existe na tabela Curso. O SGBD recusa o comando imediatamente — é a integridade referencial, definida através da FOREIGN KEY, protegendo o banco contra o famoso "registro órfão" que discutimos na Semana 8.
SELECT — a consulta mais usada de toda a carreira em dados

Se o INSERT é a porta de entrada, o SELECT é, disparado, o comando mais usado no dia a dia de qualquer pessoa que trabalha com bancos de dados — lembra do papel do analista de dados, na Semana 9, cujo trabalho gira quase inteiramente em torno dele? Sua forma mais simples consulta todas as colunas e todas as linhas de uma tabela:

SQL — SELECT básico
-- Todas as colunas, todas as linhas:
SELECT * FROM Aluno;

-- Apenas as colunas que interessam:
SELECT nome, matricula FROM Aluno;

-- Filtrando linhas com WHERE:
SELECT nome FROM Aluno WHERE curso_codigo = 1;

O * (asterisco) é um atalho útil para explorar dados rapidamente, mas evite usá-lo em aplicações reais em produção: pedir todas as colunas quando só duas ou três são necessárias desperdiça processamento e tráfego de rede — um detalhe pequeno, mas que faz diferença real em sistemas com muitos usuários simultâneos.

A cláusula WHERE é o que transforma um SELECT de "me dê tudo" para "me dê exatamente o que eu preciso" — e aceita os mesmos tipos de operadores lógicos e de comparação que você já usa em qualquer linguagem de programação:

OperadorSignificadoExemplo
=, <> (ou !=)Igual, diferenteWHERE curso_codigo = 1
>, <, >=, <=Maior, menor, maior/igual, menor/igualWHERE carga_horaria >= 60
AND, OR, NOTCombinação de condiçõesWHERE curso_codigo = 1 AND ativo = TRUE

Pare e pense: por que você acha que WHERE email = NULL não funciona como esperado em nenhum SGBD, sendo necessário usar WHERE email IS NULL? Uma dica: pense em NULL não como "um valor chamado nulo", mas como "a ausência de qualquer valor" — e no que significa comparar uma ausência com algo.

ORDER BY e DISTINCT — organizando o resultado

Duas cláusulas simples, mas extremamente usadas no dia a dia, ajudam a organizar o resultado de uma consulta: ORDER BY define a ordem das linhas retornadas, e DISTINCT remove valores duplicados do resultado.

SQL — ORDER BY e DISTINCT
-- Alunos em ordem alfabética pelo nome:
SELECT nome FROM Aluno ORDER BY nome ASC;

-- Disciplinas da maior para a menor carga horária:
SELECT nome, carga_horaria FROM Disciplina ORDER BY carga_horaria DESC;

-- Lista, sem repetição, de todos os códigos de curso
-- que já têm pelo menos um aluno matriculado:
SELECT DISTINCT curso_codigo FROM Aluno;
Funções de agregação: transformando linhas em respostas de negócio

Lembra do papel do analista de dados (Semana 9), cujo trabalho é responder perguntas como "quantos alunos estão matriculados no curso X?" ou "qual a carga horária média das disciplinas?" Esse tipo de pergunta não pede uma lista de linhas — pede um resumo calculado sobre várias linhas de uma vez. É exatamente para isso que existem as funções de agregação:

FunçãoO que calculaExemplo
COUNT()Quantidade de linhasSELECT COUNT(*) FROM Aluno;
SUM()Soma dos valores de uma coluna numéricaSELECT SUM(carga_horaria) FROM Disciplina;
AVG()Média dos valoresSELECT AVG(carga_horaria) FROM Disciplina;
MAX() / MIN()Maior e menor valorSELECT MAX(carga_horaria) FROM Disciplina;

Sozinhas, essas funções resumem a tabela inteira em um único número. A verdadeira força delas aparece quando combinadas com GROUP BY, que agrupa as linhas por uma coluna em comum antes de aplicar a agregação — respondendo perguntas do tipo "quantos alunos existem em cada curso", em vez de "quantos alunos existem no total":

SQL — GROUP BY e HAVING
-- Quantidade de alunos matriculados EM CADA curso:
SELECT curso_codigo, COUNT(*) AS total_alunos
FROM Aluno
GROUP BY curso_codigo;

-- A mesma consulta, mas mostrando só os cursos
-- com MAIS DE 20 alunos matriculados.
-- HAVING filtra depois do agrupamento (WHERE filtra antes):
SELECT curso_codigo, COUNT(*) AS total_alunos
FROM Aluno
GROUP BY curso_codigo
HAVING COUNT(*) > 20;
Resultado ilustrativo da primeira consulta
curso_codigototal_alunos
134
218
327
Diferença entre WHERE e HAVING: WHERE filtra linhas individuais antes de qualquer agrupamento acontecer; HAVING filtra os grupos já calculados, depois do GROUP BY. É por isso que não é possível escrever WHERE COUNT(*) > 20 — nesse ponto da consulta, a contagem ainda nem existe.
Um primeiro contato com JOIN: reencontrando as chaves estrangeiras

Chegamos ao momento em que todo o trabalho de modelagem da Semana 8 realmente compensa. Até aqui, consultamos cada tabela isoladamente — mas o valor real de um banco de dados relacional está em combinar informações de tabelas diferentes, exatamente através das chaves estrangeiras que projetamos naquela semana. O comando que faz essa combinação se chama JOIN.

SQL — JOIN combinando Aluno e Curso
-- Sem JOIN, "curso_codigo" é só um número sem sentido.
-- Com JOIN, trazemos o NOME do curso junto do aluno,
-- usando exatamente a chave estrangeira da Semana 8:
SELECT Aluno.nome, Curso.nome_curso
FROM Aluno
JOIN Curso ON Aluno.curso_codigo = Curso.codigo;
Resultado ilustrativo
nomenome_curso

João Pedro SantosTecnologia em Banco de Dados
Camila FerreiraAnálise e Desenvolvimento de Sistemas

A cláusula ON diz ao SGBD exatamente como as duas tabelas se conectam — normalmente, comparando a chave estrangeira de uma com a chave primária da outra, seguindo a mesma lógica de cardinalidade 1:N que estudamos na Semana 8. Este é apenas um primeiro contato com o assunto — JOIN tem variações importantes (interno, externo, entre outras) que costumam ganhar uma disciplina inteira dedicada a consultas avançadas mais adiante no curso; por hoje, o essencial é entender por que ele existe: sem ele, os dados relacionados entre tabelas diferentes ficariam permanentemente isolados uns dos outros.

UPDATE e DELETE — alterando e removendo dados com segurança
UPDATE — corrigindo e atualizando o que já existe

O comando UPDATE altera o valor de uma ou mais colunas em linhas que já existem em uma tabela. Sua sintaxe é enganosamente simples — e é exatamente essa simplicidade que exige atenção redobrada, como veremos a seguir:

SQL — UPDATE básico
-- Corrigindo o nome de um aluno específico:
UPDATE Aluno
SET nome = 'Maria Eduarda S. Oliveira'
WHERE matricula = 2026001;

-- Atualizando mais de uma coluna ao mesmo tempo:
UPDATE Curso
SET nome_curso = 'Tecnologia em Banco de Dados e IA', carga_horaria = 1700
WHERE codigo = 1;
O erro mais perigoso e mais comum do SQL: executar um UPDATE sem a cláusula WHERE não gera erro nenhum — ele simplesmente aplica a alteração a todas as linhas da tabela, de uma vez só. UPDATE Aluno SET curso_codigo = 1;, sem WHERE, transferiria instantaneamente todos os alunos do banco de dados inteiro para o curso 1 — provavelmente não é isso que você queria fazer.

Essa é a razão pela qual boa parte das ferramentas profissionais de SQL, e boa parte dos DBAs experientes (Semana 9), seguem uma disciplina pessoal simples e valiosa: escrever e testar o SELECT equivalente ao WHERE pretendido antes de rodar o UPDATE ou o DELETE — confirmando visualmente quais linhas exatamente serão afetadas, antes de efetivamente alterá-las ou removê-las.

DELETE — removendo linhas específicas

O comando DELETE remove linhas inteiras de uma tabela, sempre respeitando o filtro definido em WHERE:

SQL — DELETE básico
-- Removendo um único aluno, pela chave primária:
DELETE FROM Aluno WHERE matricula = 2026001;

-- Removendo todas as matrículas de um curso que foi encerrado:
DELETE FROM Matricula WHERE disciplina_codigo = 103;

Voltando à comparação que fizemos na Semana 15: diferente do DROP TABLE (que apaga a tabela inteira, estrutura e dados) e do TRUNCATE TABLE (que apaga todos os dados de uma vez, mas mantém a tabela), o DELETE é o único dos três comandos que permite remover exatamente um subconjunto de linhas, através do WHERE — e, dentro de uma transação ainda não confirmada (o COMMIT da TCL, mencionado na Semana 15), essa remoção ainda pode ser desfeita com um ROLLBACK.

Checklist de segurança para UPDATE e DELETE (o irmão prático do checklist de DDL)

Na Semana 15, fechamos com um checklist de segurança para comandos DDL destrutivos. A boa notícia é que a mesma disciplina se aplica, quase ponto a ponto, aos comandos UPDATE e DELETE da DML:

  • Testar o WHERE primeiro com um SELECT — confirmar visualmente exatamente quais linhas seriam afetadas antes de alterá-las ou apagá-las de fato.
  • Garantir que existe um backup recente (Semana 14) antes de qualquer alteração em massa em produção.
  • Usar transações quando o SGBD permitir — iniciar com um comando de controle, rodar o UPDATE/DELETE, conferir o resultado, e só então confirmar com COMMIT (ou desfazer com ROLLBACK caso algo esteja errado).
  • Pedir revisão de uma segunda pessoa em alterações críticas de produção, aplicando a mesma separação de funções discutida na Semana 9.
Exemplo guiado completo: um dia inteiro de manipulação de dados

Para fechar a semana, vamos simular, em sequência, o tipo de rotina que um sistema acadêmico real executa constantemente — juntando os quatro comandos estudados hoje:

SQL — INSERT, SELECT, UPDATE e DELETE em sequência
-- 1. Um novo aluno se matricula (INSERT):
INSERT INTO Aluno (matricula, nome, curso_codigo)
VALUES (2026045, 'Pedro Henrique Costa', 1);

-- 2. A secretaria confirma que o cadastro entrou corretamente (SELECT):
SELECT * FROM Aluno WHERE matricula = 2026045;

-- 3. O aluno se casa e solicita a atualização do sobrenome (UPDATE),
--    sempre testando o WHERE antes com um SELECT equivalente:
SELECT nome FROM Aluno WHERE matricula = 2026045;
UPDATE Aluno SET nome = 'Pedro Henrique Costa Lima' WHERE matricula = 2026045;

-- 4. O aluno tranca o curso e solicita o cancelamento
--    de suas matrículas em disciplinas do semestre atual (DELETE):
SELECT * FROM Matricula WHERE aluno_matricula = 2026045;
DELETE FROM Matricula WHERE aluno_matricula = 2026045;

Note como cada comando de alteração ou remoção é precedido por um SELECT equivalente — exatamente a disciplina de segurança que discutimos há pouco, aplicada na prática, do começo ao fim de um fluxo de trabalho real.
