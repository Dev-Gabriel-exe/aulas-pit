Do modelo conceitual ao modelo lógico
Retomando a Semana 7 com um objetivo novo

Na semana passada, construímos nosso primeiro diagrama ER: identificamos as entidades Aluno e Curso, seus atributos (incluindo o atributo chave sublinhado de cada uma), e o relacionamento Matricula-se entre elas, com seu próprio atributo data_matrícula. Esse diagrama é excelente para comunicar a estrutura de um sistema — para um colega de equipe, para um cliente, para qualquer pessoa que precise entender "o que o banco vai guardar" sem precisar saber nada de SQL.

Só que existe um problema prático: nenhum SGBD relacional entende um diagrama ER diretamente. O PostgreSQL, o MySQL, o SQL Server — nenhum deles "lê" retângulos, elipses e losangos. Eles entendem exatamente uma coisa: tabelas, com colunas, tipos de dados e restrições. É exatamente essa lacuna que o modelo lógico preenche: ele é o resultado de um processo sistemático e bem definido de tradução do diagrama ER para uma estrutura de tabelas.

Por que essa etapa é intermediária, e não a etapa final: o modelo lógico ainda é, tecnicamente, independente do SGBD específico que você vai usar — as regras que veremos hoje valem igualmente para PostgreSQL, MySQL, Oracle ou qualquer outro SGBD relacional. É só na etapa seguinte, o modelo físico (que veremos de forma mais aplicada a partir da Semana 15, quando estudarmos DDL), que entram detalhes específicos de cada SGBD, como tipos de dado proprietários ou parâmetros de armazenamento.
Regra 1 — toda entidade forte vira uma tabela

A primeira e mais intuitiva das regras de conversão: cada entidade do diagrama ER se transforma em uma tabela no modelo lógico, e cada atributo simples da entidade se transforma em uma coluna dessa tabela. O atributo identificado como chave na modelagem conceitual se torna a chave primária (Primary Key, ou PK) da tabela — o mecanismo que, dentro do SGBD, garante que cada linha possa ser identificada de forma única e sem ambiguidade.

Modelo conceitual (Semana 7)

Entidade ALUNO, com atributos matrícula (chave), nome e data_nasc.

Modelo lógico (esta semana)

Tabela Aluno, com colunas matricula (PRIMARY KEY), nome e data_nasc.

Note que essa tradução é praticamente direta quando a entidade só tem atributos simples. A complicação real aparece quando entram em cena os outros tipos de atributo que estudamos na semana passada — compostos, multivalorados e derivados — porque nenhum deles se traduz da mesma forma simples de "um atributo, uma coluna". É isso que vamos resolver nos próximos três cartões.

Regra 2 — atributos compostos são "achatados" em colunas separadas

Lembra do atributo composto endereço que vimos na Semana 7 — divisível em rua, número, bairro e cidade? Tabelas relacionais não têm um conceito nativo de "coluna dentro de coluna": cada coluna guarda exatamente um valor atômico. Por isso, a regra de conversão para atributos compostos é: "achatar" (flatten) cada subparte em uma coluna independente, geralmente prefixando o nome para preservar a origem.

Conversão de atributo composto — endereço
-- Atributo composto no ER: endereço (rua, número, bairro, cidade)
-- Vira QUATRO colunas independentes na tabela:

CREATE TABLE Aluno (
    matricula     INT PRIMARY KEY,
    nome          VARCHAR(100) NOT NULL,
    endereco_rua      VARCHAR(80),
    endereco_numero   VARCHAR(10),
    endereco_bairro   VARCHAR(60),
    endereco_cidade   VARCHAR(60)
);

Repare que a informação conceitual de que essas quatro colunas "pertencem" logicamente a um mesmo atributo composto se perde um pouco na tabela — ela existe apenas na convenção de nomenclatura (o prefixo endereco_). Isso é uma perda aceitável: o modelo relacional simplesmente não tem, nativamente, um conceito de atributo aninhado — e é exatamente esse tipo de estrutura aninhada que, décadas depois, bancos de documentos como o MongoDB reintroduziriam de forma nativa (como vimos na Semana 4).

Regra 3 — atributos multivalorados exigem uma tabela nova

Este é o caso mais delicado dos quatro, e também o mais importante de entender bem. Lembre do atributo telefones — um aluno pode ter vários telefones ao mesmo tempo. Por que não podemos simplesmente criar uma coluna telefones e guardar todos os números separados por vírgula dentro dela?

Tentação (errada): uma coluna com vários valores

telefones = '86999998888, 86988887777' — parece prático, mas quebra a atomicidade exigida pelo modelo relacional: fica impossível buscar "todos os alunos com um telefone específico" de forma eficiente, impossível garantir que cada telefone seja único, e a coluna vira um texto livre difícil de validar.

Solução correta: nova tabela

Criar uma tabela separada, Telefone_Aluno, com uma linha para cada telefone, e uma chave estrangeira apontando de volta para o Aluno dono daquele telefone. Cada telefone vira um dado atômico, pesquisável e validável individualmente.

Conversão de atributo multivalorado — telefones
-- A entidade original permanece exatamente como antes:
CREATE TABLE Aluno (
    matricula INT PRIMARY KEY,
    nome      VARCHAR(100) NOT NULL
);

-- O atributo multivalorado vira uma TABELA NOVA,
-- com uma linha para cada telefone e uma chave
-- estrangeira apontando para o Aluno dono dele:
CREATE TABLE Telefone_Aluno (
    id             INT PRIMARY KEY,
    numero         VARCHAR(20) NOT NULL,
    aluno_matricula INT,
    FOREIGN KEY (aluno_matricula) REFERENCES Aluno(matricula)
);

-- Agora Maria pode ter quantos telefones precisar,
-- cada um em sua própria linha, sem quebrar atomicidade.

Adiantando um pouco o vocabulário da Aula 3 desta mesma semana: essa "tabela nova apontando de volta para a tabela original" é, tecnicamente, um relacionamento 1:N entre Aluno e Telefone — um aluno pode ter vários telefones, mas cada telefone pertence a exatamente um aluno. Ou seja: um atributo multivalorado, no fundo, esconde um relacionamento 1:N implícito. Isso conecta diretamente com o assunto da Aula 2.

Regra 4 — atributos derivados normalmente não viram coluna nenhuma

Fechando o conjunto de regras desta aula: atributos derivados — como idade, calculável a partir de data_nasc — geralmente não são armazenados fisicamente no modelo lógico. A recomendação padrão é calcular o valor derivado sempre que ele for necessário, seja em uma consulta SQL, seja na camada de aplicação.

Por que essa regra existe: se você armazenasse a idade como uma coluna comum, ela ficaria desatualizada no dia seguinte ao aniversário do aluno, a menos que algum processo automático a atualizasse constantemente. Isso cria exatamente o tipo de redundância e risco de inconsistência que vimos, lá na Semana 2, como o problema histórico que os SGBDs foram criados para resolver. Existem exceções legítimas — por exemplo, quando o cálculo é muito custoso e o dado precisa ser consultado com altíssima frequência — mas elas são otimizações conscientes, feitas depois de entender o custo de manter o dado sincronizado, e não a regra geral.

Cardinalidade — decidindo onde entra a chave estrangeira
O que é cardinalidade, e por que ela é a peça que faltava

No diagrama da Semana 7, o relacionamento Matricula-se conectava Aluno e Curso — mas propositalmente deixamos uma pergunta em aberto: quantos alunos um curso pode ter, e em quantos cursos um aluno pode se matricular? Essa informação é chamada de cardinalidade do relacionamento, e ela é exatamente o dado que faltava para sabermos, com precisão, como transformar o relacionamento em algo que uma tabela consiga representar.

Existem três categorias fundamentais de cardinalidade entre duas entidades:

CardinalidadeSignificadoExemplo
1:1 (um para um)Uma instância da entidade A se relaciona com no máximo uma instância da entidade B, e vice-versa.Um Funcionário possui um único Crachá de acesso; um Crachá pertence a um único Funcionário.
1:N (um para muitos)Uma instância de A pode se relacionar com várias instâncias de B, mas cada instância de B se relaciona com apenas uma instância de A.Um Curso tem vários Alunos matriculados; cada Aluno pertence a um único Curso.
N:N (muitos para muitos)Uma instância de A pode se relacionar com várias instâncias de B, e vice-versa, simultaneamente.Um Aluno cursa várias Disciplinas; uma Disciplina é cursada por vários Alunos.
Eco da Semana 4: perceba que o problema do relacionamento N:N — que fez o modelo hierárquico "quebrar" lá na Semana 4 — é exatamente o mesmo tipo de relacionamento que vamos aprender a resolver hoje, com uma elegância que o modelo hierárquico nunca teve: uma tabela associativa.
Regra de ouro do 1:N — a chave estrangeira sempre vai para o lado "N"

Vamos usar o cenário Curso 1:N Aluno (um curso tem vários alunos; cada aluno pertence a um único curso) para fixar a regra mais importante desta aula, que vale para praticamente todo relacionamento 1:N que você vai modelar na sua carreira: a chave estrangeira é sempre adicionada na tabela do lado "N" do relacionamento, apontando para a chave primária do lado "1".

SQL — relacionamento 1:N entre Curso e Aluno
-- Lado "1": Curso não recebe nenhuma coluna nova.
CREATE TABLE Curso (
    codigo      INT PRIMARY KEY,
    nome_curso  VARCHAR(80) NOT NULL
);

-- Lado "N": Aluno recebe a CHAVE ESTRANGEIRA,
-- apontando para o curso ao qual pertence.
CREATE TABLE Aluno (
    matricula     INT PRIMARY KEY,
    nome          VARCHAR(100) NOT NULL,
    curso_codigo  INT,
    FOREIGN KEY (curso_codigo) REFERENCES Curso(codigo)
);

Por que a chave vai para o lado N, e nunca o contrário? Pense na lógica: cada linha de Aluno só pode ter um valor de curso_codigo (uma coluna guarda um único valor) — e isso é exatamente compatível com "cada aluno pertence a um único curso". Já uma linha de Curso não conseguiria guardar, em uma única coluna, uma referência para vários alunos ao mesmo tempo — o que quebraria a atomicidade que já discutimos na Aula 1. Essa é, na prática, a mesma lógica que já apareceu no exemplo do atributo multivalorado telefones: um atributo multivalorado é, disfarçadamente, um relacionamento 1:N, e por isso se resolve exatamente da mesma forma.

Relacionamento N:N — quando nenhum dos dois lados pode receber a chave sozinho

Agora o caso mais rico da aula: Aluno N:N Disciplina (um aluno cursa várias disciplinas; uma disciplina é cursada por vários alunos). Se tentássemos colocar a chave estrangeira em qualquer um dos dois lados, teríamos o mesmo problema do parágrafo anterior — nenhuma coluna simples consegue representar "vários" valores ao mesmo tempo.

A solução, que é exatamente a mesma "virada de chave" que Codd introduziu no modelo relacional (Semana 4) para resolver esse problema de forma elegante, é criar uma terceira tabela, chamada de tabela associativa (ou tabela de junção, ou tabela ponte): ela contém, no mínimo, duas chaves estrangeiras — uma para cada entidade envolvida — e a combinação das duas normalmente forma a própria chave primária dessa nova tabela.

SQL — relacionamento N:N entre Aluno e Disciplina
CREATE TABLE Aluno (
    matricula INT PRIMARY KEY,
    nome      VARCHAR(100) NOT NULL
);

CREATE TABLE Disciplina (
    codigo      INT PRIMARY KEY,
    nome        VARCHAR(80) NOT NULL
);

-- TABELA ASSOCIATIVA: representa o próprio
-- relacionamento "Aluno CURSA Disciplina".
-- A PRIMARY KEY é a COMBINAÇÃO das duas chaves
-- estrangeiras (chave primária composta).
CREATE TABLE Matricula (
    aluno_matricula   INT,
    disciplina_codigo INT,
    data_matricula    DATE,
    PRIMARY KEY (aluno_matricula, disciplina_codigo),
    FOREIGN KEY (aluno_matricula) REFERENCES Aluno(matricula),
    FOREIGN KEY (disciplina_codigo) REFERENCES Disciplina(codigo)
);

Note que a tabela Matricula também é o lugar natural para guardar o atributo data_matrícula que identificamos, na Semana 7, como pertencente ao relacionamento, e não a nenhuma das duas entidades — mais uma prova de que a modelagem conceitual e o modelo lógico conversam entre si em cada detalhe. Cada linha dessa tabela associativa representa uma matrícula específica: "a Maria cursa a disciplina de Cálculo I, desde 03/02/2026" — uma combinação única de aluno e disciplina.

O caso especial do 1:1 — quando a escolha de lado é uma decisão de projeto

O relacionamento 1:1 é o mais raro dos três na prática, mas merece atenção porque, diferente do 1:N, a escolha de qual tabela recebe a chave estrangeira não é imposta pela estrutura — é uma decisão de projeto. No exemplo Funcionário 1:1 Crachá, a chave estrangeira poderia estar tanto em Funcionário (apontando para o crachá) quanto em Crachá (apontando para o funcionário) — tecnicamente, ambas funcionam.

Critério comum de escolha

Colocar a chave estrangeira na tabela que representa a entidade "opcional" ou "dependente" do relacionamento. Se nem todo Funcionário tem Crachá (alguns ainda aguardam emissão), faz mais sentido colocar a referência em Crachá, que sempre existirá em função de um Funcionário.

Garantindo a unicidade "1"

Não basta declarar a chave estrangeira — é preciso garantir também que ela não se repita, usando uma restrição UNIQUE sobre a coluna de chave estrangeira. Sem isso, nada impediria, tecnicamente, que dois funcionários apontassem para o mesmo crachá — o que quebraria a cardinalidade 1:1 pretendida.

Chave estrangeira, integridade referencial e o algoritmo completo
O que exatamente uma chave estrangeira garante

Já usamos a chave estrangeira (foreign key, ou FK) nas duas aulas anteriores, mas vale formalizar o que ela realmente garante dentro do SGBD, e não apenas "para que serve". Uma FK é uma coluna (ou conjunto de colunas) em uma tabela que referencia a chave primária de outra tabela (ou, ocasionalmente, da mesma tabela). O SGBD usa essa referência para impor uma regra chamada integridade referencial: toda vez que uma linha tiver um valor em uma coluna de chave estrangeira, esse valor precisa existir de fato como chave primária na tabela referenciada.

Em outras palavras: o SGBD nunca permite que uma linha de Aluno aponte, através de curso_codigo, para um curso que não existe. Uma referência inválida desse tipo é chamada de registro órfão, e é exatamente esse tipo de inconsistência silenciosa que a integridade referencial existe para impedir — sem ela, seria perfeitamente possível ter alunos "presos" a cursos fantasmas, uma inconsistência praticamente indetectável até que alguém tentasse gerar um relatório e os números não fechassem.

Conectando com a Semana 5: essa é uma manifestação muito concreta daquela ideia de "instância válida × instância inválida" que estudamos ao falar de esquema e instância — a chave estrangeira é uma das restrições do esquema que filtra, em tempo real, quais instâncias o SGBD permite que existam.
O que acontece quando o registro "pai" é apagado ou alterado

Surge uma pergunta prática inevitável: se um Curso é excluído do banco de dados, o que deve acontecer com todos os Alunos que apontavam para ele através da chave estrangeira? Deixar esses alunos "órfãos" quebraria a integridade referencial que acabamos de estudar — então o SGBD precisa de uma regra explícita para essa situação, chamada de ação referencial, definida no momento em que a chave estrangeira é criada.

Ação referencialComportamento ao apagar/alterar o registro "pai"Quando costuma fazer sentido
RESTRICT (ou NO ACTION)O SGBD impede a exclusão/alteração do pai enquanto existir qualquer registro filho apontando para ele.Quando a existência do filho depende criticamente do pai e apagar o pai "por engano" seria perigoso — ex.: não permitir apagar um Curso que ainda tem alunos matriculados.
CASCADEAo apagar o pai, o SGBD apaga automaticamente todos os registros filhos relacionados (ou atualiza o valor, em caso de UPDATE CASCADE).Quando o filho não faz sentido sem o pai — ex.: apagar um Pedido e, junto, apagar automaticamente todos os seus Itens_Pedido.
SET NULLAo apagar o pai, o SGBD define automaticamente a chave estrangeira do filho como NULL, "desvinculando" o registro sem apagá-lo.Quando o filho pode existir de forma independente do pai — ex.: apagar uma Categoria de produto, mas manter os Produtos, apenas sem categoria.
SQL — declarando a ação referencial na chave estrangeira
CREATE TABLE Aluno (
    matricula    INT PRIMARY KEY,
    nome         VARCHAR(100) NOT NULL,
    curso_codigo INT,
    FOREIGN KEY (curso_codigo) REFERENCES Curso(codigo)
        ON DELETE RESTRICT
        ON UPDATE CASCADE
);
-- Leitura: "não deixe apagar um Curso que ainda tenha
-- alunos matriculados nele; mas se o código do curso
-- mudar, atualize automaticamente essa referência nos alunos."

A escolha entre RESTRICT, CASCADE e SET NULL nunca é puramente técnica — ela reflete uma decisão sobre as regras de negócio do sistema que está sendo modelado, e é comum que times de desenvolvimento debatam bastante sobre qual ação faz mais sentido para cada relacionamento específico.

O algoritmo completo de conversão ER → Relacional

Chegou o momento de juntar tudo o que vimos nas três aulas de hoje em um roteiro único, que você pode seguir sempre que precisar converter qualquer diagrama ER em um modelo lógico completo:

PassoRegra
1Toda entidade forte vira uma tabela; seu atributo chave vira a chave primária (PK) dessa tabela.
2Todo atributo simples vira uma coluna com o mesmo nome e um tipo de dado compatível.
3Todo atributo composto é "achatado": cada subparte vira uma coluna independente.
4Todo atributo multivalorado vira uma tabela nova, com chave estrangeira apontando de volta para a entidade original (relacionamento 1:N implícito).
5Atributos derivados, em geral, não viram coluna nenhuma — são calculados quando necessário.
6Para cada relacionamento 1:N, a chave estrangeira vai na tabela do lado "N", apontando para a chave primária do lado "1".
7Para cada relacionamento N:N, cria-se uma tabela associativa nova, com chaves estrangeiras para as duas entidades (compondo, normalmente, a própria chave primária) e com os atributos do relacionamento, se houver.
8Para cada relacionamento 1:1, escolhe-se um dos dois lados para receber a chave estrangeira (com restrição UNIQUE), com base em qual entidade é mais "dependente" da outra.
9Para cada chave estrangeira, define-se explicitamente a ação referencial (RESTRICT, CASCADE ou SET NULL) de acordo com a regra de negócio do sistema.
Vale memorizar para a P2: este algoritmo de 9 passos é, na prática, uma versão detalhada e aplicável do processo que Codd (Semana 4) descreveu apenas em termos matemáticos abstratos. É o elo prático entre a teoria dos conjuntos por trás do modelo relacional e o CREATE TABLE que você vai escrever a partir da Semana 15.
Exemplo guiado completo: aplicando o algoritmo do início ao fim

Vamos aplicar as 9 regras a um cenário levemente ampliado do exemplo da Semana 7, incluindo agora também a entidade Disciplina: um Aluno pertence a um único Curso (1:N), e um Aluno cursa várias Disciplinas ao mesmo tempo, sendo que cada Disciplina é cursada por vários Alunos (N:N).

SQL — modelo lógico final, resultado completo da conversão
-- Passo 1, 2 e 3: entidades fortes viram tabelas
CREATE TABLE Curso (
    codigo     INT PRIMARY KEY,
    nome_curso VARCHAR(80) NOT NULL
);

CREATE TABLE Disciplina (
    codigo      INT PRIMARY KEY,
    nome        VARCHAR(80) NOT NULL,
    carga_horaria INT
);

-- Passo 6: relacionamento 1:N (Curso -> Aluno).
-- A chave estrangeira "curso_codigo" fica no lado "N".
CREATE TABLE Aluno (
    matricula     INT PRIMARY KEY,
    nome          VARCHAR(100) NOT NULL,
    curso_codigo  INT,
    FOREIGN KEY (curso_codigo) REFERENCES Curso(codigo)
        ON DELETE RESTRICT
        ON UPDATE CASCADE
);

-- Passo 7: relacionamento N:N (Aluno <-> Disciplina).
-- Tabela associativa com chave primária composta.
CREATE TABLE Matricula (
    aluno_matricula   INT,
    disciplina_codigo INT,
    data_matricula    DATE NOT NULL,
    PRIMARY KEY (aluno_matricula, disciplina_codigo),
    FOREIGN KEY (aluno_matricula) REFERENCES Aluno(matricula)
        ON DELETE CASCADE,
    FOREIGN KEY (disciplina_codigo) REFERENCES Disciplina(codigo)
        ON DELETE RESTRICT
);

Esse é, de ponta a ponta, um modelo lógico completo e coerente: três tabelas, duas chaves estrangeiras simples e uma chave primária composta, cada decisão de cardinalidade e de ação referencial justificada pela regra de negócio (não faz sentido apagar um Curso que ainda tem alunos matriculados; mas se um Aluno for removido do sistema, é razoável que suas matrículas em disciplinas sejam removidas junto, via CASCADE).

Modelo lógico

Entidades viram tabelas; atributos simples viram colunas; compostos são achatados; multivalorados viram tabelas novas; derivados normalmente não são armazenados.

Cardinalidade

1:N → chave estrangeira no lado "N". N:N → tabela associativa com chave composta. 1:1 → escolha de projeto, com UNIQUE.

Chave estrangeira

Garante integridade referencial — impede registros órfãos. Ações RESTRICT, CASCADE e SET NULL definem o que ocorre ao apagar o registro pai.
